# 实施计划：模块 3.1 数据采集与结构化 + 3.2 核心数据总线

## 设计决策

**核心数据总线采用扁平环形数组**，而非分槽块状结构。理由：

- Analysis 线程一次消费一个 `Voltage_block`（~50ms 间隔），但**块内 ~90K 个采样点的写入操作本身是微秒级的**（仅循环赋值 + 原子更新 WriteIndex）
- 扁平结构下 WriteIndex 在块写入完成后原子跳跃推进，消费者在任何时刻读取的都是最新有效数据
- 分槽结构引入"帧边界"导致消费者在帧间隙读到重复数据，不符合"读写指针前一段"的实时语义

由于 Analysis 线程本身就有 ~50ms 的帧间隔（等待下一个 Voltage_block），消费者无论使用哪种结构都受此约束。扁平结构消除了槽边界的额外延迟。

```csharp
[StructLayout(LayoutKind.Sequential)]
public struct StructuredSample
{
    public long Timestamp;   // 采样点序号ID（全局递增）
    public long Time;        // 物理采集时间戳（Stopwatch.GetTimestamp()，单位=Stopwatch.Frequency ticks）
    public double CH1;       // 通道1原始电压值
    public double CH2;       // 通道2原始电压值
    public double Vis;       // 能见度 (m)，暂占位 0.0
    public double Cn2;       // 大气折射率结构常数 (m⁻²/³)，暂占位 0.0

    // 未来扩展数据
    public double Temp;      // 温度 (°C)
    public double Humi;      // 湿度 (%)
    public double Press;     // 气压 (hPa)
    public double WindSpd;   // 风速 (m/s)
    public double Rain;      // 降雨量 (mm)
    public double WindDir;   // 风向 (°)
}
// 总大小：12 × 8 = 96 bytes
```

⚠️ `Time` 用 `long` 存 `Stopwatch.GetTimestamp()` 值。**每个采样点拥有独立物理时间戳**——见步骤 4 中的关键设计。

### 步骤 2：改造 RawRingBuffer → 核心数据总线 `CoreDataBus`（扁平环形数组）

**改造文件**: `ConsoleApp1/Service/SharedMemoryClient.cs` — 基于现有 `RawRingBuffer` 类重写

#### 2.1 共享内存布局

```
┌─────────────────────────────────────┐
│ CoreBusHeader (固定大小)             │  ← basePtr + 0
│   WriteIndex  long   写指针（单调递增）│
│   ChannelCount int  通道数 (2)       │
│   BufferLength int  环形缓冲区容量    │
│   SampleRate  int   采样率 (Hz)      │
├─────────────────────────────────────┤
│ 数据区                              │  ← dataPtr (basePtr + headerSize)
│   StructuredSample[0]               │
│   StructuredSample[1]               │
│   ...                               │
│   StructuredSample[BufferLength-1]  │
└─────────────────────────────────────┘
```

#### 2.2 头结构（复用 RawMemoryHeader 布局，直接改名）

```csharp
[StructLayout(LayoutKind.Sequential)]
public struct CoreBusHeader
{
    public long WriteIndex;    // 写指针（单调递增，取模 = 真实索引）
    public int ChannelCount;   // 通道数
    public int BufferLength;   // 环形缓冲区容量（采样点数）
    public int SampleRate;     // 采样率 (Hz)
}
```

与现有 `RawMemoryHeader` 布局完全一致，仅改名，保证向后兼容。

#### 2.3 方法接口

| 方法 | 调用方 | 说明 |
|------|--------|------|
| `Create(int channels, int bufferLength, int sampleRate)` | WebAPI 启动 | 创建共享内存 |
| `Open()` | ConsoleApp1 / WebAPI 消费进程 | 连接已创建的共享内存 |
| `Write(ref StructuredSample sample)` | Analysis 线程 | **逐条流式写入**：写一个采样点，立即推进 WriteIndex（单写者无锁，内存屏障保证可见性） |
| `ReadLatestSingle(out StructuredSample sample)` | 持久化/低频UI（WebAPI） | 读取写指针前 1 条（单条，定时调用） |

#### 2.3.1 Write 逐条流式写入实现

```csharp
// 单写者场景（Analysis 线程独占写入），无需锁，内存屏障保证可见性
public void Write(ref StructuredSample sample)
{
    long index = header->WriteIndex;
    long pos = index % BufferLength;
    dataPtr[pos] = sample;                    // 步骤1：写数据

    Interlocked.MemoryBarrier();              // 步骤2：内存屏障——确保步骤1对消费者可见
    
    header->WriteIndex = index + 1;           // 步骤3：推进 WriteIndex（消费者此时可见新数据）
    // 注意：此处用普通赋值而非 Interlocked.Increment，
    // 因为只有 Analysis 一个写入者，不存在并发写竞争
}
```

**为什么不用 `Interlocked.Increment`**：单写者场景下，写操作不需要原子交换（CAS），仅需保证写入顺序。`Interlocked.MemoryBarrier()` 确保步骤 1 的数据在步骤 3 的 WriteIndex 更新之前对其他 CPU 核心可见。消费者读到新 WriteIndex 时，对应位置的数据一定已经写入完成。

#### 2.4 消费者读取模式

| 消费者 | 运行位置 | 数据来源 | 触发方式 | 间隔 |
|--------|----------|----------|----------|------|
| 持久化线程 | WebAPI（后续实现） | CoreDataBus.ReadLatestSingle() | 定时轮询 | 1s/5s/30s/1min/5min |
| 低频 UI 线程 | WebAPI（后续实现） | CoreDataBus.ReadLatestSingle() | 定时轮询 | 7s |
| 检测线程 | ConsoleApp1（子进程） | DetectionChannel（`StructuredSample[]`） | Channel 事件驱动 | 每帧（~50ms） |

数据流：
```
Analysis → CoreDataBus → {持久化, 低频UI}（跨进程 MMF）
Analysis → DetectionChannel → 检测线程（同进程 Channel，直接传 StructuredSample[]）
```

检测线程**不接触 CoreDataBus**，分析线程将结构化数据分两路：一路写总线，一路通过 Channel 直接传给检测线程。

#### 2.5 缓冲区容量与内存计算

- `BufferLength = 10,000,000`（10 秒 × 1MHz 缓冲，适配持久化/低频 UI 的读取间隔）
- 数据区：10,000,000 × 96 bytes ≈ **915 MB**
- 头结构：24 bytes（可忽略）
- 总共享内存 ≈ **915 MB**（DAQ 工控机 ≥ 8GB RAM 可接受）

### 步骤 4：完善 Analysis 线程（逐条流式写入 + 分两路输出）

**改造文件**: `ConsoleApp1/Service/AD_Controlcs.cs` 的 `Analysis()` 方法（line 708-770）

Analysis 线程将同一个 `batch[]` 分两路输出，写入方式有本质区别：

| 路径 | 目标 | 写入方式 | 消费者读取方式 |
|------|------|----------|---------------|
| 路 1 | CoreDataBus | **逐条**：每个 StructuredSample 独立写入环形缓冲区，可被单条访问 | `ReadLatestSingle()`（持久化/低频 UI 按定时逐条取） |
| 路 2 | DetectionChannel | **整批**：整个 StructuredSample[] 数组一次性写入 Channel | `Channel.Reader.ReadAsync()`（检测线程按帧整批处理） |

```csharp
private unsafe void Analysis()
{
    // 1. 连接核心数据总线
    var coreBus = new CoreDataBus();
    coreBus.Open();

    // 2. 预分配结构化数组（复用，无 GC）
    //    最大帧大小 ≈ 92160（单通道），充分预分配 200000
    StructuredSample[] batch = new StructuredSample[200_000];
    long globalSampleIndex = 0;

    // 3. 预计算每采样点 Stopwatch ticks 增量（恒定值）
    //    Stopwatch.Frequency ≈ 10MHz，1MHz 采样 → 10 ticks/sample
    double ticksPerSample_d = (double)Stopwatch.Frequency / Program.deviceconfig.SampleRate;
    long ticksPerSample = (long)Math.Round(ticksPerSample_d);

    Voltage_block voltageBlock = default;
    try
    {
        while (ADDataTest_RunFlag)
        {
            try
            {
                voltageBlock = Analysischannel.Reader.ReadAsync(cts.Token)
                                          .AsTask().GetAwaiter().GetResult();
            }
            catch (OperationCanceledException) { break; }

            int count = (int)voltageBlock.SampleCount;
            long baseTick = voltageBlock.StartTick; // ADDraw 捕获的高精度起始时间戳

            // 逐采样点填充结构化数据——每个采样点拥有独立物理时间戳
            for (int i = 0; i < count; i++)
            {
                batch[i].Timestamp = globalSampleIndex++;
                batch[i].Time      = baseTick + i * ticksPerSample;
                batch[i].CH1       = voltageBlock.Voltage1?[i] ?? 0;
                batch[i].CH2       = voltageBlock.Voltage2?[i] ?? 0;
                batch[i].Vis       = 0.0;  // TODO: 接入 VisibilityCalculator
                batch[i].Cn2       = 0.0;  // TODO: 接入 Cn2ProfileInverter
                batch[i].Temp      = 0.0;
                batch[i].Humi      = 0.0;
                batch[i].Press     = 0.0;
                batch[i].WindSpd   = 0.0;
                batch[i].Rain      = 0.0;
                batch[i].WindDir   = 0.0;

                // ===== 路 1：逐条流式写入 CoreDataBus（立即置于总线可视区）=====
                coreBus.Write(ref batch[i]);
            }

            // ==================== 路 2：整批写入 DetectionChannel ====================
            var detArr = ArrayPool<StructuredSample>.Shared.Rent(count);
            Array.Copy(batch, 0, detArr, 0, count);
            if (!DetectionCh.Writer.TryWrite(detArr))
                ArrayPool<StructuredSample>.Shared.Return(detArr);

            // 归还数组池
            if (voltageBlock.Voltage1 != null)
                ArrayPool<double>.Shared.Return(voltageBlock.Voltage1);
            if (voltageBlock.Voltage2 != null)
                ArrayPool<double>.Shared.Return(voltageBlock.Voltage2);
        }
    }
    catch (Exception ex)
    {
        ADDataTest_RunFlag = false;
        cts.Cancel();
        init();
        GrpcClient.SendErrorMessage("分析数据线程出现异常: " + ex.Message);
    }
    finally { coreBus?.Dispose(); }
}
```

**关键设计——每个采样点独立物理时间戳**：

ADDraw 线程已在 `AD_Controlcs.cs:553` 捕获 `startTick = Stopwatch.GetTimestamp()` 并存入 `voltageBlock.StartTick`。Analysis 线程据此为每个采样点计算独立时间戳：

```
sample[i].Time = startTick + i × (Stopwatch.Frequency / SampleRate)
```

例如 1MHz 采样率、Stopwatch.Frequency ≈ 10MHz：
- `ticksPerSample ≈ 10`
- `sample[0].Time = startTick + 0`
- `sample[1].Time = startTick + 10`
- `sample[999999].Time = startTick + 9999990`（≈ 1 秒后）

这样写入环形缓冲区的采样点具有**严格单调递增且间隔均匀的时间戳**。消费者定时读取时，两条相邻采样的 `Time` 差值天然等于期望间隔，与消费者自身的调度抖动无关。

### 步骤 4.5：新建检测线程（第 5 线程）

检测线程由数据采集子进程维护。Analysis 线程将结构化数据**分两路**：一路写入 CoreDataBus，一路通过 `DetectionChannel` 直接传给检测线程（同进程 Channel，传递 `StructuredSample[]` 真数据）。

#### 4.5.1 DetectionChannel（第 4 个 Channel）

在 `AD_Controlcs.cs` 的 `CreateNewDataChannel()` 方法中新增：

```csharp
public static Channel<StructuredSample[]> DetectionCh;

DetectionCh = Channel.CreateBounded<StructuredSample[]>(
    new BoundedChannelOptions(8)
    {
        FullMode = BoundedChannelFullMode.DropOldest,
        SingleWriter = true,
        SingleReader = true,
        AllowSynchronousContinuations = true
    },
    dropped =>
    {
        // 背压丢弃时归还数组，防止泄漏
        if (dropped != null)
            ArrayPool<StructuredSample>.Shared.Return(dropped);
    });
```

Channel 中直接传 `StructuredSample[]` 数组，其 `.Length` 即本帧采样点数。时间戳从数组首尾元素的 `Time` 字段获取。

#### 4.5.2 Analysis 线程分两路写入（已在步骤 4 中实现，此处说明架构）

```csharp
// 路 1：逐条流式写入 CoreDataBus（在 for 循环内每填充一个样本立即写入）
// for (int i = 0; i < count; i++) { ...; coreBus.Write(ref batch[i]); }

// 路 2：整批写入 DetectionChannel（租用 → 拷贝 → 写入）
var detArr = ArrayPool<StructuredSample>.Shared.Rent(count);
Array.Copy(batch, 0, detArr, 0, count);
if (!DetectionCh.Writer.TryWrite(detArr))
    ArrayPool<StructuredSample>.Shared.Return(detArr);
```

两路并行写入，互不阻塞。CoreDataBus 逐条写入支撑持久化/低频 UI 的单条读取；DetectionChannel 整批写入支撑检测线程的帧级处理。

数组大小即 `count`，检测线程通过 `.Length` 获取采样点数，通过 `detArr[0].Time` / `detArr[count-1].Time` 获取起止时间。

#### 4.5.3 检测线程方法（第 5 个线程）

检测线程从 Channel 直接接收完整的 `StructuredSample[]` 数组，无需访问 CoreDataBus：

```csharp
private void Detection()
{
    while (ADDataTest_RunFlag)
    {
        StructuredSample[] samples;
        try
        {
            samples = DetectionCh.Reader.ReadAsync(cts.Token)
                                   .AsTask().GetAwaiter().GetResult();
        }
        catch (OperationCanceledException) { break; }

        // === 检测分析（轻量级阈值比较，<1ms/帧） ===
        for (int i = 0; i < samples.Length; i++)
        {
            var s = samples[i];
            // TODO: 后续接入 SignalQualityDetector / ConditionAssessor
            if (s.CH1 < 0.01 && s.CH1 > -0.01 && i > 100)
            {
                GrpcClient.SendErrorMessage(
                    $"通道1信号遮挡 @序号{s.Timestamp}", "SIGNAL_OBSTRUCTION");
                break;
            }
        }

        // 用完归还数组
        ArrayPool<StructuredSample>.Shared.Return(samples);
    }
}
```

```
检测线程
  → GrpcClient.SendErrorMessage(msg, "SIGNAL_OBSTRUCTION")
    → ResponseQueue.Enqueue(AdResponse{ MessageType="Error", ErrorCode="SIGNAL_OBSTRUCTION" })

        → WebAPI GrpcServiceImpl:173 (case "Error")
```

后续可新增专用 `detection_alarm` MessageType，当前先用 `Error` 复用现有通道。

## 关键设计点

1. **每个采样点独立物理时间戳** — `sample[i].Time = startTick + i * (Frequency / SampleRate)`，严格单调递增、间隔均匀
2. **消费者只读单条** — 持久化/低频 UI 线程：`Sleep(interval)` → `ReadLatestSingle()` → 写入/发布
3. **检测线程通道传真数据** — Analysis 分两路：一路写 CoreDataBus，一路通过 `Channel<StructuredSample[]>` 直接传数组副本。检测线程无需接触 CoreDataBus
4. **路1逐条流式写入** — Analysis 每处理一个采样点立即写入 CoreDataBus，WriteIndex 逐 1 推进，消费者无需等待整批处理完毕即可读取新数据。内存屏障保证可见性
5. **CoreDataBus 与 UISharedBuffer 完全物理隔离**（`DAQ_CORE_DATA_BUS` vs `DAQ_UI_RING_BUFFER`）
6. **不改变现有 4 线程 + 3 Channel 的进程内数据流**，新增第 5 线程 + 第 4 Channel
7. **`long` 存储时间戳** — 规避 `DateTime` 跨进程 LayoutKind.Auto 歧义
8. **Vis/Cn² 占位 0.0**，完整算法见 `开发记录/大气检测激光雷达数据解析链路 — 实施计划.md`
