# 1. 概述

## 1.1 项目背景

本系统为高频数据采集设备（如1MHz采样率采集卡）构建一个高可靠、低延迟的数据处理与分发体系，实现从原始数据采集、结构化处理，到持久化存储、双路UI数据分发、实时数据检测判别的全流程闭环，适配前端不同场景的数据展示与异常告警需求。

## 1.2 设计目标

- 模块解耦：采集、存储、分发、检测模块独立运行，单模块故障不影响其他链路

- 高效通信：基于环形缓冲区+内存映射文件实现零拷贝进程间通信，适配高频数据场景，保障实时性

- 双路分发：支持高频实时数据与低频归档数据的差异化推送，满足前端多场景展示

- 稳定推送：以独立主控进程托管MQTT客户端，保障数据推送的可靠性

- 实时检测：数据检测模块通过专用的DetectionChannel（同进程Channel）接收结构化数据，告警经gRTC双向流传递至主控进程发布MQTT

# 2. 系统整体架构概览

系统分为五层，各层通过**核心数据总线（环形缓冲区+内存映射文件）**与**UI专用缓冲区（MMF）**实现双隔离数据流转，无跨模块耦合，其中数据检测层通过独立的 DetectionChannel 接入：

1. **数据采集与结构化层**：原始数据采集、预处理与标准化转换，Analysis线程分两路输出

2. **核心数据总线层**：基于环形缓冲区的内存映射文件，作为持久化与低频UI模块的数据枢纽，采用"单写多读"模式，读取端读取写入位置前一段有效数据；检测线程不经过核心数据总线，由Analysis线程通过独立的DetectionChannel直接供给数据

3. **数据持久化层**：独立线程从CoreDataBus读取单条数据，实现周期的数据归档存储

4. **数据分发层**：分为低频UI链路与高频实时链路，通过MQTT向订阅端推送数据；高频链路由主控进程托管客户端服务，使用独立UI专用缓冲区，不占用核心数据总线资源

5. **数据检测与工况判别层**：由ConsoleApp1子进程维护的独立线程，通过DetectionChannel接收结构化数据，告警经gRPC双向流传递至主控进程发布MQTT告警Topic

# 3. 模块详细设计

## 3.1 数据采集与结构化模块

### 模块组成：ADWork → ADDraw → Analysis

- **ADWork**：与硬件采集设备交互，读取原始二进制采样数据

- **ADDraw**：原始数据预处理，包括去噪、通道对齐、数据缓存，输出原始电压值数据，直接供给UI刷新线程；同时捕获高精度采集起始时间戳 `startTick = Stopwatch.GetTimestamp()`，存入 `Voltage_block.StartTick`

- **Analysis**：结构化转换，将原始电压值解析为标准化 `StructuredSample` 格式（12字段，96 bytes），包含：

  - `Timestamp`：采样点序号ID（全局递增），用于数据完整性追踪
  - `Time`：物理采集时间戳（Stopwatch ticks），每个采样点独立计算：`sample[i].Time = startTick + i * (Stopwatch.Frequency / SampleRate)`，严格单调递增且间隔均匀
  - `CH1/CH2`：通道原始电压值
  - `Vis`：能见度，衡量大气透明度。暂占位 0.0，后续接入 VisibilityCalculator
  - `Cn2`：大气折射率结构常数。暂占位 0.0，后续接入 Cn2ProfileInverter
  - `Temp/Humi/Press/WindSpd/Rain/WindDir`：未来扩展气象字段，暂占位 0.0

**核心：逐条流式写入 + 分两路输出**

Analysis 线程每填充完一个 StructuredSample，立即执行两路操作：

1. **路 1（逐条流式写入 CoreDataBus）**：每处理完一个采样点，立即通过 `CoreDataBus.Write(ref sample)` 写入环形缓冲区。内部三步：写数据到缓冲区 -> `Interlocked.MemoryBarrier()` 内存屏障 -> 推进 WriteIndex。WriteIndex 逐1推进，消费者无需等待整批写完即可读取新数据

2. **路 2（整批写入 DetectionChannel）**：处理完整个 Voltage_block 后，通过 `Channel<StructuredSample[]>` 将本帧全部结构化数据一次性传递给检测线程（同进程 Channel，避免跨 MMF 读取开销）

**输出流向**：
- 路 1（CoreDataBus）：-> 持久化线程（按定时单条读取） + 低频 UI 线程（按定时单条读取）
- 路 2（DetectionChannel）：-> 检测线程（按帧整批处理，不接触 CoreDataBus）

## 3.2 核心数据总线（环形缓冲区+内存映射文件）

- **实现方式**：基于操作系统原生内存映射文件机制，构建全局共享的扁平环形数据缓冲区，共享内存名称 `DAQ_CORE_DATA_BUS`，头结构 `CoreBusHeader`（WriteIndex/ChannelCount/BufferLength/SampleRate），数据区为 `StructuredSample[BufferLength]`

- **核心职责**：

  - 作为持久化与低频UI两个消费者的数据枢纽，提供统一的单条读取接口

  - 实现零拷贝数据共享，避免进程间数据拷贝开销，适配1MHz高频数据场景，保障实时性

  - 采用"单写多读"模式——Analysis 线程独占写入、持久化/低频UI消费者各自读取最新数据，无需加锁

  - 防止数据覆盖：`BufferLength = 10,000,000`（10秒 x 1MHz 缓冲，约915 MB），适配持久化最慢5分钟读取间隔

- **读写规则**：

  - 写入端（Analysis 线程）：**逐条流式写入**——每处理完一个采样点立即调用 `Write(ref StructuredSample sample)`，三步写入：写数据到环形缓冲区 -> `Interlocked.MemoryBarrier()` 内存屏障 -> 推进 `WriteIndex`。WriteIndex 逐1推进，WriteIndex永远不会回绕（取模才对应真正缓冲区位置），消费者无需等待整批写完即可读取最新数据

  - 读取端（持久化/低频UI线程）：通过 `ReadLatestSingle(out StructuredSample sample)` 读取写指针前1条最新数据（"写指针前一段"规则）。消费者按自身周期定时调用，无需维护独立读指针，读取到的采样点 `Time` 字段天然反映精确采集时刻

  - 检测线程不经过核心数据总线（由 DetectionChannel 直接供给），避免跨进程 MMF 读取延迟

- **访问方式**：跨进程模块通过映射同一 MMF 文件到进程虚拟地址空间直接读写，访问延迟低至微秒级。ConsoleApp1 侧 Analysis 调用 `Open()` 连接主进程（WebAPI）创建的共享内存

## 3.3 数据持久化线程

- **核心职责**：从核心数据总线（环形缓冲区）读取结构化数据，按配置周期写入CSV文件实现归档存储

- **关键逻辑**：

  - 支持多周期读取配置：1s/5s/30s/1min/5min，可通过配置文件调整。读取方式：按配置周期调用 `ReadLatestSingle()` 读取最新单条结构化数据，该采样点的 `Time` 字段精确反映物理采集时刻，时序间隔天然等于配置周期

  - 独立线程运行，不阻塞采集与分发、检测主流程

  - 采用**批量写入+异步刷盘**策略，降低磁盘IO压力，提升写入性能

**输出**：CSV格式的归档数据文件，支持离线回溯与分析

## 3.4 低频UI数据分发链路

- **模块组成**：UI显示线程 -> MQTT客户端（低频Topic）

- **核心职责**：为前端提供周期更新的低频数据，用于慢变趋势、统计数据类图表展示

- **关键逻辑**：

  - 读取频率：每7秒从核心数据总线调用一次 `ReadLatestSingle()`，获取最新单条结构化数据

  - 数据推送：将数据发布至MQTT服务器的低频数据专属Topic

  - 前后端解耦：数据更新逻辑由前端控制，后端仅按周期推送数据，不参与前端展示逻辑

**下游对接**：前端订阅低频Topic，实现7秒周期的图表更新

## 3.5 高频实时数据分发链路

- **模块组成**：UI刷新线程（降采样处理） -> UI专用缓冲区（独立MMF） -> 主控进程 -> MQTT客户端（实时Topic）

- **核心职责**：为前端提供低延迟的降采样实时数据，用于动态波形、快速变化数据类图表展示

- **关键逻辑**：

  - 数据读取：UI刷新线程直接读取ADDraw输出的原始电压值数据，不经过核心数据总线、不读取Analysis结构化数据

  - 降采样处理：对原始电压值执行降采样（降采样比1000），降低数据传输量，避免前端渲染压力

  - 数据写入：降采样后的实时数据写入独立UI专用缓冲区（MMF），该缓冲区与核心数据总线完全隔离，仅服务于高频实时链路

  - 主控进程职责：

    - 仅从UI专用缓冲区读取降采样实时数据，不访问核心数据总线

    - 托管MQTT客户端服务，实现自动重连、断连缓存，保障推送稳定性

    - 将数据发布至MQTT服务器的实时数据专属Topic

**下游对接**：前端订阅实时Topic，实现低延迟实时图表刷新

## 3.6 数据检测与工况判别模块

- **模块定位**：独立后台高频线程，与采集、分发、持久化模块完全解耦。运行于ConsoleApp1（数据采集子进程），通过DetectionChannel接收Analysis输出的结构化数据，无需访问CoreDataBus

- **核心职责**：从DetectionChannel接收分析线程输出的整批 `StructuredSample[]` 数组，逐帧完成业务层智能分析与状态判别，输出工况结果与异常告警

- **检测分析内容**：

  - 环境指标判别：基于结构化数据中的Vis、Cn2参数，完成能见度数值高低阈值判定、折射率实时分析，输出工况等级

  - 信号质量判别：通过分析CH1/CH2通道数据，识别信号遮挡状态、检测信号噪声含量、判断波形畸变异常，定位信号故障

  - 工况异常判定：识别数据超限、跳变、断连、数据畸变等异常状态，触发告警逻辑

- **关键逻辑**：

  - 读取策略：检测线程在 `Channel<StructuredSample[]>.Reader.ReadAsync()` 上阻塞等待，Analysis写完一帧后通过Channel唤醒，无需轮训、零CPU空转

  - 实时性保障：采用轻量级CPU运算（阈值对比、数值运算、特征判定），单帧数据处理耗时<1ms，端到端总延迟约1ms（Analysis完成写入 -> DetectionChannel传递 -> 检测线程接收 -> 检测分析 -> 告警推送）

  - 线程优先级：`AboveNormal`，高于UI/持久化线程（`Normal`），低于采集线程（`Highest`），确保采集流程不被阻塞，同时保障检测的实时响应

- **数据输出流向**：

  - 本地侧：记录异常日志、保存工况状态快照，便于故障回溯

  - 推送侧：检测线程通过 `GrpcClient.SendErrorMessage(msg, alarmCode)` 将工况判别结果、异常事件发送至现有gRPC双向流消息队列；主控进程WebAPI收到后发布至MQTT告警专属Topic，与实时、低频Topic隔离

  - 前端侧：前端订阅告警Topic，实现工况状态指示灯、异常弹窗、故障事件列表展示，支持异常详情查看

# 4. 关键数据流向说明

## 4.1 数据采集与总线写入流程

1. ADWork读取硬件原始数据 -> ADDraw预处理（输出原始电压值，供给UI刷新线程；捕获 startTick） -> Analysis结构化转换（填充StructuredSample的12个字段；每个采样点独立时间戳）

2. Analysis分两路输出：
   - 路1：每次填充完一个采样点，立即 `CoreDataBus.Write(ref sample)` 逐条流式写入（三步：写数据 -> 内存屏障 -> 推进WriteIndex）
   - 路2：整帧处理完毕后，`ArrayPool` 租用数组 + 拷贝 + 写入 `DetectionChannel`（Channel传递 `StructuredSample[]` 真数据）

3. 核心数据总线读取端（持久化、低频UI线程）均通过 `ReadLatestSingle()` 读取写指针前1条最新数据，各自按定时独立调用

## 4.2 数据持久化存储流程

1. 数据持久化线程按配置周期（1s/5s/30s/1min/5min）从核心数据总线调用 `ReadLatestSingle()` 读取最新单条结构化数据

2. 批量写入CSV文件，异步刷盘实现数据落盘

## 4.3 低频UI数据分发流程

1. UI显示线程每7秒从核心数据总线调用 `ReadLatestSingle()` 读取最新单条结构化数据

2. 将数据发布至MQTT服务器的低频数据Topic

3. 前端订阅低频Topic，按7秒周期更新对应图表

## 4.4 高频实时数据分发流程

1. UI刷新线程直接读取ADDraw输出的原始电压值数据，不经过核心数据总线

2. 对原始电压值执行1000倍降采样处理，降低数据量

3. 降采样后的实时数据写入独立UI专用缓冲区（MMF）

4. 主控进程从UI专用缓冲区读取降采样数据，通过托管的MQTT客户端发布至实时数据Topic

5. 前端订阅实时Topic，实现低延迟实时图表刷新

## 4.5 数据检测与告警推送流程（新版）

1. Analysis线程处理完一帧（整个 Voltage_block）后，从 `ArrayPool` 租用数组、拷贝 `StructuredSample[]` 整批写入 `DetectionChannel`

2. 检测线程在 `DetectionChannel.Reader.ReadAsync()` 上阻塞等待，收到数据后逐帧执行信号质量检测、工况智能判别，生成工况结果与异常信息

3. 检测线程通过 `GrpcClient.SendErrorMessage(msg, alarmCode)` 将告警发送至现有gRPC双向流消息队列

4. WebAPI GrpcServiceImpl 收到 `MessageType="Error"` 的响应后，调用 `_mqttEventPublisher.PublishStateChangedAsync(...)` 发布至MQTT告警专属Topic

5. 前端订阅告警Topic，展示工况状态、触发异常弹窗，记录故障事件

# 5. 核心设计要点说明

## 5.1 统一IPC机制：环形缓冲区+内存映射文件（实时性核心）

系统跨进程通信采用"环形缓冲区+内存映射文件"实现，无多机制适配成本，天生适配高频实时场景：

- 零拷贝数据共享，避免管道、消息队列的拷贝开销，访问延迟低至微秒级，适配1MHz高频采集需求

- 单写多读设计：Analysis线程独占写入，持久化/低频UI独立读取最新数据，无锁阻塞，最大化实时性

- 数据安全：`BufferLength = 10,000,000` 适配采集率，确保未读取数据不被新数据覆盖

- 无额外部署依赖，所有模块仅需访问同一文件路径即可接入系统

## 5.2 DetectionChannel：同进程Channel直传结构化数据

检测线程的数据不经过CoreDataBus，通过同进程 `Channel<StructuredSample[]>` 直接接收：

- 消除跨进程MMF读取的微秒级延迟，检测数据处理完全在ConsoleApp1单进程内完成，延迟<1us

- Channel传递的是 `StructuredSample[]` 数组的引用（极轻量），数据备份从 `ArrayPool` 租用，检测线程用完归还

- 事件驱动：检测线程在 `ReadAsync()` 上阻塞等待，不需要轮询CoreDataBus，零CPU空转

- 零proto改动：检测告警通过现有 `GrpcClient.SendErrorMessage()` 进入gRPC双向流，WebAPI端 `GrpcServiceImpl` 使用已有的 `MessageType="Error"` 分支处理

## 5.3 双路MQTT分发+告警推送的职责划分（三层Topic隔离）

|链路类型|数据内容|更新频率|MQTT Topic定位|前端应用场景|
|---|---|---|---|---|
|低频UI链路|单条结构化数据（ReadLatestSingle）|7s/次|低频数据Topic|慢变趋势、统计数据、历史趋势图表|
|高频实时链路|降采样后的原始电压数据|近实时|实时数据Topic|实时波形、快速变化的动态数据图表|
|数据检测告警链路|工况结果、异常告警信息|近实时（每帧触发）|告警专属Topic|工况状态展示、异常弹窗、故障事件列表|
前后端职责解耦：后端仅负责按链路特性推送数据，前端通过订阅不同Topic实现差异化展示，更新逻辑、告警触发逻辑完全由前端控制

## 5.4 主控进程的角色定位

主控进程并非传统Web服务器，而是高频实时链路的核心调度节点：

- 核心职责：仅从UI专用缓冲区读取降采样实时数据，托管MQTT客户端服务、保障实时数据推送的稳定性，不访问核心数据总线

- 告警中转职责：通过gRPC双向流接收ConsoleApp1子进程上报的检测告警，转发至MQTT告警专属Topic

- 优势：避免了用Web服务器承载高频推送的性能瓶颈，同时实现MQTT客户端的可靠运行（如自动重连、断连缓存）

## 5.5 数据检测模块的实时性保障

数据检测模块的实时性由DetectionChannel的事件驱动机制保证，无任何性能瓶颈：

- 读取实时性：Channel的 `ReadAsync()` 在Analysis写入后立即唤醒检测线程，数据传递延迟<1us
- 处理实时性：轻量级运算，单帧处理耗时<1ms，端到端总延迟约1ms
- 解耦不阻塞：独立线程运行，仅消费DetectionChannel，与采集线程无交互，不影响采集效率

## 5.6 读写分离与链路解耦设计

- 采集、持久化、低频分发、高频分发、数据检测模块完全独立运行，互不阻塞

- 各模块访问各自的数据源：持久化/低频UI访问CoreDataBus（MMF）、高频链路访问UISharedBuffer（MMF）、检测线程访问DetectionChannel（Channel）

- 单模块故障（如持久化线程崩溃、检测线程异常）不影响其他链路的正常运行，提升系统稳定性

# 7. 优化与扩展建议

## 7.1 核心数据总线优化

- 优化环形缓冲区大小：根据采集数据包大小、采集率，动态调整缓冲区容量，进一步降低内存占用，同时避免数据覆盖

- 当前 `Write(ref sample)` 使用普通赋值 + MemoryBarrier，单写者场景下性能最优；若未来改为多写入者，可升级为 Interlocked.Increment

## 7.2 MQTT链路优化

- 采用分层Topic命名规范（如 sensor/lowfreq/device01、sensor/realtime/device01、sensor/alarm/device01），方便订阅方区分数据类型，便于后期扩展多设备采集

- 主控进程增加断连缓存机制，MQTT服务器断连时缓存少量实时数据与告警信息，恢复连接后补发，避免数据丢失

- 为不同Topic配置合适的QoS等级：低频数据、告警信息使用QoS 1保障可靠交付，实时数据使用QoS 0保障低延迟

## 7.3 持久化存储优化

- 支持按时间分片存储CSV文件（如按小时/天生成新文件），避免单个文件过大，便于离线分析与管理

- 引入压缩存储（如CSV压缩为ZIP），降低磁盘占用

- 支持数据过滤写入，可配置仅存储关键数据、异常数据或指定周期数据，减少存储压力

## 7.4 数据检测模块优化

- 支持检测阈值可配置：能见度、折射率、噪声的判定阈值可通过配置文件调整，无需修改代码，适配不同应用场景

- 增加异常分级告警：根据异常严重程度，分为一般告警、严重告警，前端可差异化展示（如不同颜色弹窗）

- 增加检测日志持久化：将工况判别结果、异常事件详细日志写入本地文件，便于故障回溯与问题定位

## 7.5 可扩展性扩展

- 新增模块（如报警模块、边缘计算模块）可直接通过访问核心数据总线或新增 Channel 接入系统，无需修改现有模块逻辑

- 支持多设备采集扩展，核心数据总线与UI专用缓冲区均可按设备划分独立数据段，实现多设备数据隔离，检测、分发链路可同步扩展

- 前端可扩展多终端适配（如PC端、移动端），仅需订阅对应MQTT Topic，无需修改后端逻辑
