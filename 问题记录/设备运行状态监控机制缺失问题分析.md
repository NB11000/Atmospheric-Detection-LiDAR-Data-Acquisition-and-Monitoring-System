# 设备运行状态监控机制缺失问题分析

> 文档版本：v2.0
> 创建时间：2026-05-10

> 状态：已完成分析，待实施

---

## 1. 问题陈述

主控进程（WebAPI）缺少一个统一的**非命令链路状态变更**的监控与事件发布模块。

### 1.1 设计前提：强乐观 UI + 命令链路正确性

系统的前后端交互遵循以下架构约定：

- **MQTT RPC 命令链路保障正确性**：前端通过 MQTT RPC 发出命令后，一旦收到"命令已受理"的确认，即乐观更新 UI 至目标状态。后端在状态缓存中更新后，**不需要再通过 `events/state_changed` 主题广播此次变更**——因为命令响应本身就是确认载体，状态变更的正确性内置于命令链路中。
- **命令链路之外的变更依赖后端监控主动发布**：当状态变更不由 MQTT RPC 命令触发时（如设备异常断开、硬件故障、检测告警、激光器意外离线等），没有命令响应可以作为"确认载体"，此时后端必须主动通过 `events/state_changed` 主题广播，确保多客户端状态的最终一致性。

```
状态变更的两种路径：

  [A] MQTT RPC 命令链路（已知，正确性有保障）
      MQTT RPC 命令 → gRPC → 子进程 → command_response
      → SystemStateService 更新缓存 → 响应 RPC 调用方
      → 调用方已收到确认，无需 state_changed 广播

  [B] 命令链路之外（未知，需要后端主动探测并广播）
      设备异常断开 / 硬件故障 / 检测告警 / 激光器意外离线 / ...
      → SystemStateService 更新缓存
      → 必须主动广播 state_changed，否则多客户端状态不一致
```

### 1.2 核心症状

当前系统对 **[B] 类变更** 的监控和发布机制存在以下问题：

- **路径 [B] 的 6 类场景中，MQTT 推送覆盖不完整**：激光器状态变更在 MQTT 上完全不可见（代码被注释）；检测告警不触发 `state_changed`；`[STATE]` 结构化上报已设计但未实现
- **推送逻辑分散在 4 个不同模块中**，覆盖不一致、行为不对称（有的推 MQTT 不推 SignalR，有的推 SignalR 不推 MQTT）
- **没有主动健康评估机制**，只能被动等待错误上报或 MQTT Will Message

## 2. 状态变更分类与现状追踪

### 2.1 分类准则

| 分类 | 触发源 | 是否需要 MQTT state_changed 广播 | 原因 |
|---|---|---|---|
| **[A] 命令链路** | MQTT RPC / HTTP API 发出的 gRPC 命令得到成功响应 | **不需要** | 命令响应已携带确认，调用方乐观更新，正确性在命令链路中保障 |
| **[B] 非命令链路** | 设备异常断开、线程崩溃、检测告警、激光器意外离线、MQTT 断连等 | **必须** | 无命令响应作为确认载体，多客户端需通过广播感知状态变化 |

### 2.2 路径 [A]：命令链路变更（设计上不需要广播）

| 场景 | 触发方式 | 状态更新位置 | 是否需要广播 | 当前行为 |
|---|---|---|---|---|
| OPEN_DEVICE 成功 | MQTT RPC / HTTP | `GrpcServiceImpl:414-423` | ❌ 不需要 | 无广播 ✓ |
| CLOSE_DEVICE 成功 | MQTT RPC / HTTP | `GrpcServiceImpl:426-435` | ❌ 不需要 | 无广播 ✓ |
| START_AD 成功 | MQTT RPC / HTTP | `GrpcServiceImpl:437-446` | ❌ 不需要（但 AcquiringStateChanged 内部事件仍需） | 内部事件触发 ✓ |
| STOP_AD 成功 | MQTT RPC / HTTP | `GrpcServiceImpl:448-456` | ❌ 不需要（但 AcquiringStateChanged 内部事件仍需） | 内部事件触发 ✓ |
| 激光串口连接（通过 RPC） | MQTT RPC laser-connect | `CniLaser:86` | ❌ 不需要 | 无广播 ✓ |
| 激光串口断开（通过 RPC） | MQTT RPC laser-disconnect | `CniLaser:115` | ❌ 不需要 | 无广播 ✓ |
| 激光开启（通过 RPC） | MQTT RPC laser-on | `CniLaser:264` | ❌ 不需要 | 无广播 ✓ |
| 激光关闭（通过 RPC） | MQTT RPC laser-off | `CniLaser:281` | ❌ 不需要 | 无广播 ✓ |

**结论：路径 [A] 的 8 种场景当前行为完全正确，无需改动。**

### 2.3 路径 [B]：非命令链路变更（必须广播，当前覆盖不完整）

| 场景 | 触发方式 | 状态更新位置 | 需要广播 | 当前 MQTT | 当前 SignalR |
|---|---|---|---|---|---|
| 采集子进程 gRPC 连接 | gRPC 双向流建立 | `GrpcServiceImpl:129` | ✅ 必须 | ✓ `:139`（fire-and-forget） | ✗ 无 |
| 采集子进程 gRPC 断开 | gRPC 流异常/结束 | `GrpcServiceImpl:365` | ✅ 必须 | ✓ `:368`（fire-and-forget） | ✓ `:375` |
| 设备异常断开 (DEVICE_DISCONNECTED) | 子进程错误上报 | `GrpcServiceImpl:522-530` | ✅ 必须 | ✓ `:196`（通用Error类型） | ✓ `:203` |
| 采集异常终止 (ACQUISITION_FAILED) | 子进程错误上报 | `GrpcServiceImpl:533-542` | ✅ 必须 | ✓ `:196`（通用Error类型） | ✓ `:203` |
| 设备打开失败 (DEVICE_OPEN_FAILED) | 子进程错误上报 | `GrpcServiceImpl:545-554` | ✅ 必须 | ✓ `:196`（通用Error类型） | ✓ `:203` |
| **激光器意外断开（非RPC）** | **硬件物理断开、断电** | **无主动检测** | ✅ 必须 | ✗ **MISSING** | ✗ **MISSING** |
| **检测告警触发状态降级** | **Detection线程 gRPC 上报** | **无** | ✅ 必须 | ✗ **MISSING**（仅走 `detection/alerts` 主题） | ✗ **MISSING** |
| **MQTT 连接恢复** | MQTT 重连成功 | `SystemStateService:74` | ✅ 必须（补偿通知） | ✗ **MISSING** | ✗ **MISSING** |
| **子进程僵死/无响应** | **无主动检测** | **无** | ✅ 必须 | ✗ **MISSING** | ✗ **MISSING** |

**关键发现：路径 [B] 的 9 种场景中，仅 5 种有 MQTT 覆盖（且其中 1 种 laser 被注释），4 种完全缺失。**

## 3. 根本原因分析

### 3.1 架构层面：SystemStateService 是浅模块

当前 `SystemStateService` 只管理状态的"值"，不管理状态的"变更通知"。路径 [B] 的变更发生时，是否需要广播、广播到哪个通道，完全由调用方自行决定。这导致：

- **调用方必须知道"这是路径 [B] 变更，需要推送"这个隐含约定**
- **不同调用方对约定的理解不一致**（GrpcServiceImpl 推了 MQTT 但漏了 SignalR；CniLaser 推了 SignalR 但 MQTT 被注释掉）
- **新增路径 [B] 变更时，开发者容易遗漏推送逻辑**

实际上，现有的推送调用之所以"部分工作"，仅仅是因为开发者在 `GrpcServiceImpl` 中手动添加了推送代码——这是靠开发者记忆维持的约定，不是靠模块接口保证的约束。

### 3.2 代码层面：路径 [B] 推送调用分布

```
GrpcServiceImpl（路径B覆盖：连接/断开/错误）
    ├── 连接时：推 MQTT ✓  推 SignalR ✗
    ├── 断开时：推 MQTT ✓  推 SignalR ✓
    └── 错误时：推 MQTT ✓  推 SignalR ✓（但 eventType 为通用 "Error"）

CniLaser（路径B覆盖：完全缺失）
    ├── MQTT 推送：全部被注释掉 ✗
    └── SignalR 推送：仅 PublishLaserStateChangedAsync，且只覆盖主动操作场景

SystemStateService（路径B覆盖：MQTT断连无广播）
    └── 仅触发内部事件 AcquiringStateChanged / MqttConnectionStateChanged

DetectionPublisherService（路径B覆盖：告警与状态割裂）
    └── 告警 → detection/alerts ✓ 但 → state_changed ✗
```

### 3.3 激光器非 RPC 断开的监控盲区

激光器的当前实现（`CniLaser`）完全依赖串口操作返回的 bool 值判断状态。以下场景无法检测：

1. **USB 转串口线缆物理拔出**——`SerialPort.IsOpen` 在 Windows 上可能仍返回 true
2. **激光器设备断电**——串口芯片可能仍保持连接状态
3. **操作系统挂起/恢复**——串口资源可能被回收但 CniLaser 不知情

而且，即使能检测到断开，`CniLaser.cs` 中的 MQTT 推送全部被注释掉（`:89`/`:99`/`:117`），即使未来添加主动监控，事件也发布不出去。

### 3.4 告警与状态变更是两条断开的路径

`DetectionPublisherService` 将告警发布到 `daq/{id}/detection/alerts` 主题，但从不触发 `events/state_changed`。远端监控系统需要同时订阅两个主题并自行关联。`SystemStateDto` 中没有任何反映"设备是否处于告警状态"的字段。

### 3.5 预留的 `[STATE]` 上报机制未实现

`GrpcServiceImpl.UpdateStateFromDataReport`（`:477-508`）中预留了 `[STATE]` 前缀的结构化状态消息解析逻辑，但整段代码被注释。子进程无法主动推送结构化状态，主控进程只能通过命令响应和错误消息间接推断——这对于路径 [B] 的实时监控来说是不够的。

### 3.6 主动健康评估完全缺失

| 现有机制 | 类型 | 触发条件 | 能否覆盖路径[B] |
|---|---|---|---|
| gRPC 命令 10s 超时 | 被动 | 仅当有命令发出时 | 不能——路径 [B] 没人发命令 |
| 错误消息上报 | 被动 | 仅当子进程线程崩溃时 | 部分——无法检测僵死（线程还在但不工作） |
| MQTT Will Message | 被动 | 仅当 MQTT 连接断开时 | 部分——仅覆盖主进程整体崩溃 |
| PING/PONG | 手动 | 需外部通过 MQTT RPC 调用 | 不能——需要人工/外部触发 |
| MonitorParentProcess | 单向 | 仅子进程监控父进程 | 不能——覆盖方向相反 |

**没有任何周期性机制评估：子进程是否僵死？设备是否长时间无数据？采集线程是否卡住？**

## 4. 影响范围

### 4.1 直接影响

| 影响 | 严重程度 | 说明 |
|---|---|---|
| 激光器非命令链路状态变更在 MQTT 不可见 | **高** | 远端无法感知激光器意外断开/恢复等事件 |
| 告警不触发 state_changed | **高** | 订阅方需跨 `detection/alerts` 和 `events/state_changed` 两个主题自行关联 |
| 没有主动健康评估 | **高** | 子进程僵死、设备无数据等故障无法及时发现 |
| 路径 [B] 推送覆盖不一致 | **中** | MQTT 和 SignalR 覆盖不对称，部分场景只推一个通道 |
| `[STATE]` 结构化上报缺失 | **中** | 子进程内部状态（线程状态、队列深度）在主控进程不可见 |

### 4.2 间接影响

- **调试困难**：路径 [B] 问题追踪需要在 4 个模块之间跳转
- **扩展阻力**：新增设备类型时，需要在各自模块中重复实现路径 [B] 推送逻辑
- **测试困难**：推送逻辑与业务逻辑耦合，无法独立测试路径 [B] 的对外发布行为

### 4.3 影响文件

| 文件 | 角色 | 受影响程度 |
|---|---|---|
| `WebAPI/Service/SystemStateService.cs` | 状态缓存，路径[B]发布能力缺失 | **核心** |
| `WebAPI/Service/CniLaser.cs` | MQTT推送全部注释，非RPC断开不可检测 | **高** |
| `WebAPI/Service/GrpcServiceImpl.cs` | 路径[B]推送逻辑分散且不一致 | **高** |
| `WebAPI/Service/DetectionPublisherService.cs` | 告警与state_changed路径割裂 | **中** |
| `WebAPI/Service/MqttEventPublisher.cs` | 推送实现被零散调用 | **中** |
| `WebAPI/Service/SignalRHubPublisher.cs` | 推送覆盖与MQTT不对称 | **低** |
| `ConsoleApp1/Service/AD_Controlcs.cs` | 子进程侧缺少[STATE]上报 | **低** |

## 5. 架构层面总结

### 5.1 删除测试

想象删除 `SystemStateService`。路径 [B] 状态管理的复杂度不会消失——它会分散到 `GrpcServiceImpl`、`CniLaser`、`DetectionPublisherService` 等 N 个调用方，每个调用方都需要自行管理状态值 + 自行判断是否路径 [B] + 自行推送。

**`SystemStateService` 正在做有价值的工作（状态缓存），但接口太浅——管理了"值"，没有管理"非命令链路变更的通知"。**

### 5.2 核心矛盾

路径 [A]（命令链路）和路径 [B]（非命令链路）的状态变更走的是**同一条代码路径**（都调用 `UpdateCollectorState` / `UpdateLaserState`），但它们的**发布需求完全不同**：

- 路径 [A]：调用方是 MQTT RPC Handler，命令响应已返回确认，不需要广播
- 路径 [B]：调用方是 GrpcServiceImpl（gRPC事件）/ CniLaser（硬件事件）/ DetectionPublisherService（告警事件），没有命令响应作为确认载体，必须广播

**当前 `SystemStateService` 的接口不区分这两种路径**，所以它无法在内部自动决定是否发布。这个决策被推到了调用方——而调用方并不总是做出正确的决定。

### 5.3 解决方案方向

`SystemStateService` 需要区分两种更新路径：

- **静默更新（路径 [A]）**：仅更新缓存，不广播。供 MQTT RPC Handler 在命令成功后调用。
- **广播更新（路径 [B]）**：更新缓存 + 自动广播到 MQTT + SignalR。供异常事件（gRPC 断连、硬件故障、检测告警等）调用。

这样，路径 [B] 的发布逻辑就从"调用方手动记得调"变成了"模块接口强制保证"。
