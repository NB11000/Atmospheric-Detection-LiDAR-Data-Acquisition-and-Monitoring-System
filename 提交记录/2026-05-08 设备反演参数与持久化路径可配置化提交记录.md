## 一、背景（Background）

系统已有 CaptureCard（采集卡）和 Radar（激光器）两个 RPC 可读写配置，均存储在 `DeviceConfig.json`，通过 `ConfigHelper` 统一管理。但以下两类配置仍处于不可读写状态：

### 问题

#### 1. 设备反演参数（LidarAlgorithm）不可通过 RPC 读写

`DeviceConfig.json` 中 `LidarAlgorithm` 段包含 11 个反演算法参数（增益系数、K 常数、接收孔径、滑动窗口帧数、Fernald 边界距离等），但 WebAPI 侧无对应模型、无 Options 绑定、无配置读写方法、无 RPC 端点。参数修改需手动编辑 JSON 文件并重启服务。

#### 2. 数据持久化路径不可通过 RPC 读写

上一轮提交已将 `DataDirectory` 从硬编码抽取为 `PersistenceSettings`，但配置存储在 `appsettings.json`，与 `DeviceConfig.json` 中其他设备配置管理路径不统一，且同样缺乏 RPC 读写端点。

#### 3. ConfigHelper 读写模式不统一

旧方法 `ReadDeviceConfig()`/`ReadRadarDeviceConfig()` 逐字段拷贝到 `Program` 静态属性，新方案需保持一致，避免返回 `IOptionsMonitor` 内部引用。

#### 4. 持久化路径无输入校验

`PersistenceSettings.DataDirectory` 可接受空字符串、非法字符、Windows 保留名（CON/NUL/COM1 等），导致 `PersistenceService` 运行时崩溃。

---

## 二、解决方案（Solution）

### 整体思路

统一配置管理架构：将 `LidarAlgorithm` 和 `Persistence` 配置纳入 `ConfigHelper` → `DeviceConfig.json` 单一管理链路，新增 `ConfigHandler` 作为统一配置 RPC Handler，逐字段拷贝模式与既有 CollectorHandler/LaserHandler 对齐。

### 具体实施

#### 1. 新增 LidarAlgorithmConfig 模型

`WebAPI/Models/LidarAlgorithmConfig.cs` — 11 个反演算法参数，含默认值。

#### 2. 新增 ConfigHandler 统一配置 RPC Handler

`WebAPI/MqttRpc/ConfigHandler.cs` — 4 个端点：
- `lidar-config-read` / `lidar-config-update`
- `persistence-config-read` / `persistence-config-update`

#### 3. ConfigHelper 扩展

- 注入 `IOptionsMonitor<LidarAlgorithmConfig>` 和 `IOptionsMonitor<PersistenceSettings>`
- `ReadLidarConfig()` / `ReadPersistenceConfig()` — 逐字段拷贝到 `Program.LidarConfig` / `Program.PersistenceConfig`
- `WriteLidarConfig()` / `WritePersistenceConfig()` — 读写完整 RootConfig 后写回磁盘
- 提取通用 `ReadRootConfig()` / `WriteRootConfig()` 辅助方法
- `ValidatePersistenceConfig()` — 路径校验：非空、非法字符（排除分隔符和盘符）、Windows 保留名（仅校验末段）
- `RootConfig` 扩展 `LidarAlgorithm` 和 `Persistence` 节点
- `ConfigJsonContext` 扩展序列化支持

#### 4. 配置源统一

- `appsettings.json` 删除 `Persistence` 段
- `DeviceConfig.json` 新增 `Persistence` 段，补全 `LidarAlgorithm` 11 字段（新增 `SampleRateHz`、`BlindZoneDistance_m`）
- `Program.cs`：Persistence 绑定源从 `appsettings.json` 改为 `DeviceConfig.json`，新增 `LidarAlgorithmConfig` 绑定，新增 `LidarConfig`/`PersistenceConfig` 静态属性，启动时执行 `ReadLidarConfig()`/`ReadPersistenceConfig()`
- `MqttRpcBackgroundService`：路由表合并 `ConfigHandler`

#### 5. 单元测试

`WebAPI.Tests/Config/ConfigModuleTests.cs` — 44 个用例，3 层：
- L1 模型测试：默认值、全/缺字段反序列化、多余字段忽略、零值/大数序列化
- L1 校验测试：null/空白/非法字符/保留名/合法路径 26 边界
- L3 RPC 协议测试：空 payload、畸形 JSON、序列化往返

## 五、架构影响

```
DeviceConfig.json (唯一配置源)
  ├── CaptureCard    → CollectorHandler  (collector-config-read/update)
  ├── Radar          → LaserHandler      (laser-config-read/update)
  ├── LidarAlgorithm → ConfigHandler     (lidar-config-read/update)        ← 新增
  └── Persistence    → ConfigHandler     (persistence-config-read/update)  ← 新增

ConfigHelper (统一读写)
  ├── ReadDeviceConfig()/WriteCaptureCardConfig()
  ├── ReadRadarDeviceConfig()/WriteRadarConfig()
  ├── ReadLidarConfig()/WriteLidarConfig()          ← 新增
  ├── ReadPersistenceConfig()/WritePersistenceConfig() ← 新增
  ├── ReadRootConfig()/WriteRootConfig()             ← 新增（通用辅助）
  └── ValidatePersistenceConfig()                    ← 新增（输入校验）

Program (全局静态配置实体)
  ├── CurrentConfig      : CaptureCardConfig
  ├── RadarConfig        : RadarConfig
  ├── LidarConfig        : LidarAlgorithmConfig      ← 新增
  └── PersistenceConfig  : PersistenceSettings       ← 新增
```

| 维度 | 变更前 | 变更后 |
|------|--------|--------|
| LidarAlgorithm 可读写 | 不可通过 RPC 修改 | `lidar-config-read/update` |
| Persistence 可读写 | 不可通过 RPC 修改 | `persistence-config-read/update` |
| Persistence 配置源 | appsettings.json | DeviceConfig.json（统一） |
| Persistence 输入校验 | 无 | 空白/非法字符/Windows 保留名 |
| 配置 RPC Handler | 4 个（Collector/Laser/System/Log） | 5 个（+ConfigHandler） |
| LidarAlgorithm JSON 字段 | 9 个（缺 SampleRateHz、BlindZoneDistance_m） | 11 个（补全） |
