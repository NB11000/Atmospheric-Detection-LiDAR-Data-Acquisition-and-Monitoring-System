## 三、背景（Background）

在 2026-05-04 的提交 `596e75d` 中，CoreDataBus 的 API 已稳定（`TryReadLatestSingle()` + 时间校准基准注入），CONTEXT.md 建立了一致的领域术语体系。然而，工程层面存在三个结构性缺口：

1. **缺乏自动化测试**：CoreDataBus、DetectionChannel、结构化填充逻辑、UTC 时间还原均为核心数据通路，但没有任何自动化测试覆盖。跨进程 MMF 的 `LayoutKind.Sequential` 布局一致性、环形缓冲区取模正确性、空数据返回 `false` 的语义正确性均无保障。
2. **任务管理无结构化工作流**：16 个待实施模块（时间校准、持久化 CSV、低频 MQTT 发布、检测算法、Vis/Cn² 反演、DI 重构、生命周期协调器等）分散在实施计划中，无统一的优先级、依赖关系和验收标准追踪机制。
3. **工具函数与生产代码耦合**：Analysis 线程内的 `StructuredSample` 填充逻辑和 `TimeHelper` UTC 还原逻辑尚未抽取为独立可测试的纯函数，阻碍单元测试编写。

同时，上一个提交中未完成的架构决策需要正式记录为 ADR，Agent 协作规范（Issue Tracker 格式、Triage Labels 语义）需要文档化以支撑后续 Issue 驱动开发。

## 四、问题（Problem）

### 1. 无测试覆盖核心数据通路

CoreDataBus 的以下关键行为无自动化验证：
- `Create()` 后 `ReferenceTick` / `ReferenceUtcTicks` 是否正确初始化（`> 0`）
- `Write()` → `TryReadLatestSingle()` 往返数据一致性
- `WriteIndex == 0` 时空总线返回 `false`（而非 zeroed-default 歧义）
- `BufferLength + 1` 次写入后环形取模正确
- `Open()` 后跨实例指针映射一致性

DetectionChannel 的数组所有权路径（正常传递、DropOldest 归还、TryWrite 失败归还、Drain 清空）同样无验证，`ArrayPool.Return` 的泄漏风险仅凭人工审查。

### 2. 任务分散、无结构化管理

16 个待实施模块中：
- 各自的优先级、依赖关系和阻塞关系未显式建模
- 验收标准未书面化，实施后验证依赖人脑记忆
- 每个模块的完成状态（done / ready-for-agent / ready-for-human）无统一追踪
- Agent 协作场景下缺少标准的 Issue 格式和标签体系

### 3. 工具函数内联在生产代码中

Analysis 线程循环体内，`StructuredSample` 的 12 字段赋值逻辑内联在 `AD_Controlcs.cs` 中，与 Channel 操作、CoreDataBus 写入混杂，无法独立测试时间戳单调性、Timestamp 跨帧连续性和占位字段正确性。

`TimeHelper` 作为纯函数尚未创建，但后续持久化 CSV 和 MQTT 发布均需将 `StructuredSample.Time`（Stopwatch tick）还原为 UTC 绝对时间戳。

### 4. 事件驱动生命周期的架构决策未文档化

三个消费者（波形发布、持久化、低频发布）的启停方式从"各自订阅事件"收敛为"Coordinator 集中驱动"的架构决策仅存在于口头讨论中，缺少正式的 ADR 记录，替代方案的推演和拒绝理由不可追溯。

#### 2. 工具函数提取（2 文件，+70 行）

**ConsoleApp1/Tools/TimeHelper.cs**:

```csharp
public static class TimeHelper
{
    /// <summary>
    /// 将采样时钟转换为UTC时间
    /// </summary>
    public static DateTime ToUtcDateTime(
        long sampleTick,
        long referenceTick,
        long referenceUtcTicks,
        long frequency)
    {
        long elapsedTicks = sampleTick - referenceTick;
        long elapsed100ns = elapsedTicks * 10_000_000L / frequency;
        return new DateTime(referenceUtcTicks + elapsed100ns, DateTimeKind.Utc);
    }
}
```

- 纯函数，无副作用，无需 DI
- 兼容 Windows（Stopwatch.Frequency ≈ 10MHz）和 Linux（Stopwatch.Frequency ≈ 1GHz）
- 使用 `DateTimeKind.Utc` 避免时区歧义
- 整数运算无浮点精度损失

**ConsoleApp1/Tools/SampleFiller.cs**:

```csharp
public static class SampleFiller
{
    public static long ComputeTicksPerSample(long frequency, int sampleRateKhz)
    {
        double d = (double)frequency / (double)(sampleRateKhz * 1000);
        return (long)Math.Round(d);
    }

    public static void FillOne(ref StructuredSample sample,
        int indexInFrame, long baseTick, long ticksPerSample,
        ref long globalSampleIndex, double[]? voltage1, double[]? voltage2)
    {
        sample.Timestamp = globalSampleIndex++;
        sample.Time = baseTick + indexInFrame * ticksPerSample;
        sample.CH1 = voltage1?[indexInFrame] ?? 0;
        sample.CH2 = voltage2?[indexInFrame] ?? 0;
        sample.Vis = 0.0;
        sample.Cn2 = 0.0;
        sample.Temp = 0.0;
        sample.Humi = 0.0;
        sample.Press = 0.0;
        sample.WindSpd = 0.0;
        sample.Rain = 0.0;
        sample.WindDir = 0.0;
    }
}
```

- `ComputeTicksPerSample`：将频率和 kHz 采样率转为每采样点的 Stopwatch 周期数
- `FillOne`：填充全部 12 字段，通过 `ref long globalSampleIndex` 确保跨帧连续递增
- `double[]?` 可空参数支持单通道采集模式（对应字段填 0）

| 术语 | 定义 | Avoid |

**Key Decisions 新增**:

> **11. 消费者生命周期由事件驱动，不由信号驱动**：消费者是抽样快照（7s ~ 5min 周期），非全量消费，跨进程信号唤醒每秒百万次无意义；采集状态和 MQTT 连接状态统一由 SystemStateService 发布，Coordinator 集中判断分发

- 新增 Coordinator→采集绑定服务的启停链路
- 明确波形发布和低频发布需 MQTT 连接；持久化仅需采集状态

#### 7. 架构决策记录（1 文件，+18 行）

**docs/adr/0001-event-driven-consumer-lifecycle.md**: 记录消费者生命周期管理的架构决策。

- **采纳方案**: 事件驱动，Coordinator 集中判断 `CanRun` 公式
- **拒绝方案 1**: 信号驱动（EventWaitHandle）——每秒 100 万次无意义唤醒
- **拒绝方案 2**: BackgroundService + ExecuteAsync 空实现——反模式
- **拒绝方案 3**: 各服务各自订阅 SystemStateService——逻辑分散、难以统一管理
- **后果**: SystemStateService 新增 MQTT 连接状态事件；三个服务注册为纯 Singleton；Coordinator 通过 `IEnumerable<IAcquisitionBoundService>` 发现服务
