## 一、背景（Background）

在真实 MQTT 模式下，采集启动后存在两个严重缺陷：RPC 响应超时和低频数据刷新延迟逐渐拉长。根因是 QoS 0 波形数据（每 100ms 推送 2×8KB）与控制/低频 QoS 1 消息共用同一 MQTT 连接，MQTTnet 内部对出站报文串行化，QoS 0 洪流填满 TCP 发送缓冲区后，QoS 1 消息被无限期排队。此外，keepalive 超时导致 MQTT 掉线后不自重连，因为断连原因码 `NormalDisconnection` 被误当作主动断连。

### 问题（Problem）

#### 1. QoS 0 波形洪流阻塞 QoS 1 控制通道

波形数据以 100ms 间隔发布 CH1+CH2 共 16KB 数据（QoS 0），RPC 响应和低频数据走 QoS 1，全部经由同一 MQTT 连接。MQTTnet 内部出站报文串行化导致 QoS 1 消息被排在 QoS 0 队列后面，RPC 响应超过前端 10 秒超时，低频数据刷新从 7 秒固定间隔逐渐延长到一分钟以上。

#### 2. Keepalive 超时后不自重连

MQTTnet 的 keepalive PING 超时触发断连，原因码为 `NormalDisconnection`。重连逻辑中 `OnDisconnectedAsync` 对该原因码做了早期返回，将其等同于用户主动调用 `DisconnectAsync` 的断连，导致掉线后自动重连被跳过。

#### 3. 波形发布 Await 加剧积压

`WaveformPublishService.RunLoopAsync` 中对 `PublishWaveformDataAsync` 做了 `await`——QoS 0 本身不需要等待 PUBACK，但 `await` 使定时循环等待 TCP 写完成，导致每帧耗时累积，进一步加剧通道拥堵。

### 问题（Problem）

- 前端 RPC 超时（start 正常，stop 超时，且随时间恶化）
- 前端 Cn²/Vis 图表刷新间隔非固定 7 秒，逐渐拉长
- MQTT 掉线后不自动恢复

## 二、解决方案（Solution）

### 整体思路

为波形数据建立**独立的 MQTT 连接**，与控制/低频通道物理隔离。同时移除无效的 `NormalDisconnection` 判据修复重连逻辑，移除波形发布的 `await` 让定时循环不阻塞。

### 具体实施

#### 1. MqttEventPublisher 增加波形专用客户端属性

```csharp
public IMqttClient? WaveformMqttClient { get; set; }
```

`PublishWaveformDataAsync` 改用 `WaveformMqttClient` 并通过 `_ = Task.WhenAll(...)` 发射后立刻返回，不等待 TCP 写完成。

#### 2. MqttRpcBackgroundService 管理第二条连接

- 创建波形客户端：`ClientId = {MachineId}-waveform`，`CleanSession = true`，无遗嘱消息、无主题订阅
- 新增 `ConnectWaveformAsync`：共享主连接的 TLS/auth 配置
- 新增 `OnWaveformDisconnectedAsync`：简单重连循环（3s 固定间隔，不设上限）
- `StopAsync` 中增加波形客户端的 Disconnect + Dispose 清理

#### 3. WaveformPublishService 移除 Await

```csharp
// 变更前
await _mqttEventPublisher.PublishWaveformDataAsync(...);
// 变更后
_mqttEventPublisher.PublishWaveformDataAsync(...);
```

#### 4. 修复重连逻辑

删除 `OnDisconnectedAsync` 中对 `MqttClientDisconnectReason.NormalDisconnection` 的早期返回。服务正常关闭时已通过 `_shouldReconnect = false`（在 `StopAsync` 中设置）提前退出重连循环。
