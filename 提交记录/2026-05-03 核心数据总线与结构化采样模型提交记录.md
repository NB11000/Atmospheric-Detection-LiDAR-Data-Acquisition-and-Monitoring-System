## 三、背景（Background）

数据采集子系统原有 4 线程 + 3 Channel 的进程内流水线：`ADWork → channel → ADDraw → Analysischannel → Analysis`。Analysis 线程此前为空壳（仅归还 `Voltage_block` 中的数组），无下游数据消费。系统缺少一个跨进程、结构化的采样数据分发中心。

同时，原始采样数据需支持两类消费者场景：
- **低频定时消费**（持久化 1s/5s/30s、低频 UI 7s）：仅在需要时读取单个最新采样点，不需逐帧处理
- **帧级实时消费**（检测线程 ~50ms/帧）：需要整帧采样数据进行阈值比较

两类消费者对数据粒度的需求不同，需要在同一套数据总线上分别支撑。

## 四、问题（Problem）

### 1. 缺少结构化数据模型

旧链路中采样数据以 `double[]` 数组形式流转，仅存原始电压值，缺少采样点唯一标识（`Timestamp`）、物理采集时间戳（`Time`）、以及未来扩展字段（Vis/Cn²/气象参数等）。消费者无法追溯单个采样点的完整上下文。

### 2. 缺少跨进程数据总线

已有 `UISharedBuffer`（降采样波形专用）和 `RawRingBuffer`（`double*`，已实现但未使用）。`RawRingBuffer` 按通道分槽存储 `double` 值，WriteIndex 按采样点数取模——消费者只能按长度读取，无法按"最新一条"语义访问，不适合低频定时消费场景。

### 3. Analysis 线程无下游输出

Analysis 线程仅执行数组归还，数据在此中断。结构化数据的写入（CoreDataBus）和帧级分发（DetectionChannel）均未实现。

### 4. `ArrayPool<Rent>` 尾部零值导致误报

`ArrayPool<T>.Shared.Rent(count)` 返回的数组实际长度可能大于 `count`（按 2 的幂次取整，如 92160 → 131072），多出的元素为零值。检测线程用 `samples.Length` 遍历时，尾部零值恰好满足 `CH1 < 0.01 && CH1 > -0.01` 的遮挡判断条件，触发 `SIGNAL_OBSTRUCTION` 误报。

## 五、解决方案（Solution）

### 整体思路

**扁平环形数组 + 结构化数据模型 + 两路分流输出** —— 用单写者无锁环形数组作为跨进程总线，每个采样点携带完整元数据；Analysis 线程同时向两条路径输出，分别适配帧级检测和单条定时消费。

### 具体实施

#### 1. 新建 `StructuredSample` 数据模型（+24 行）

```csharp
[StructLayout(LayoutKind.Sequential)]
public struct StructuredSample  // 12 × 8 = 96 bytes
```

跨进程 MMF 安全布局，包含采样点序号 (`Timestamp`)、物理时间戳 (`Time`)、双通道原始电压 (`CH1`/`CH2`)、以及 8 个占位扩展字段（Vis/Cn²/Temp/Humi/Press/WindSpd/Rain/WindDir）。

#### 2. `RawRingBuffer` → `CoreDataBus`（扁平环形数组）

**内存布局**：

```
┌──────────────────────────────────┐
│ CoreBusHeader (24 bytes)          │  ← basePtr + 0
│   WriteIndex    long  写指针      │
│   ChannelCount  int   通道数 (2)  │
│   BufferLength  int   容量 (10M)  │
│   SampleRate    int   采样率      │
├──────────────────────────────────┤
│ StructuredSample[0]               │  ← dataPtr
│ StructuredSample[1]               │
│ ...                               │
│ StructuredSample[BufferLength-1]  │
└──────────────────────────────────┘
```

**核心差异**：

| 维度 | RawRingBuffer（旧） | CoreDataBus（新） |
|------|---------------------|-------------------|
| 数据指针 | `double*`（单值） | `StructuredSample*`（96B 结构体） |
| WriteIndex 语义 | 取模循环 | 单调递增（消费者取模） |
| 写入粒度 | `Write(double[])` 批量 | `Write(ref StructuredSample)` 逐条流式 |
| 读取接口 | `ReadLatest(double[], int length)` 按长度 | `ReadLatestSingle()` 按最新 1 条 |
| MapName | `DAQ_RAW_BUFFER` | `DAQ_CORE_DATA_BUS` |

**逐条流式写入**（`Write`）：

```csharp
dataPtr[pos] = sample;              // 步骤1：写数据（96B 值拷贝）
Interlocked.MemoryBarrier();        // 步骤2：内存屏障，确保步骤1对消费者可见
header->WriteIndex = index + 1;     // 步骤3：推进 WriteIndex（单写者，普通赋值）
```

内存屏障保证消费者读到新 WriteIndex 时对应位置的数据一定已写入完成。

#### 3. `DetectionBatch` 消除 Rent 尾部误报

```csharp
public struct DetectionBatch
{
    public StructuredSample[] Samples;
    public int Count;  // 实际有效采样点数
}
```

检测线程遍历 `i < batch.Count` 而非 `samples.Length`，杜绝尾部零值落入检测逻辑。

#### 4. Analysis 线程两路输出

```
                    ┌── 路 1：逐条流式 ──→ CoreDataBus (MMF) ──→ 持久化/低频UI (单条取)
Analysis ──┤
                    └── 路 2：整批写入 ──→ DetectionChannel ──→ Detection 线程 (整批取)
                                      (new DetectionBatch{Samples,Count})
```

- **路 1**：for 循环内每填充一个 `StructuredSample` 立即 `coreBus.Write(ref detArr[i])`，WriteIndex 逐 1 推进，消费者无需等待整批处理完毕
- **路 2**：for 循环结束后一次性 `TryWrite(new DetectionBatch{...})`，检测线程获取完整帧

#### 5. 新增 Detection 线程（第 5 线程）

```mermaid
flowchart LR
    A[Analysis 线程] -->|DetectionChannel| B[Detection 线程]
    B -->|CH1 遮挡检测| C{CH1 ≈ 0 ?}
    C -->|是| D[GrpcClient.SendErrorMessage]
    D --> E[WebAPI GrpcServiceImpl]
    E --> F[MqttEventPublisher]
    F --> G[MQTT: daq/{MachineId}/lidar/alarm]
```

线程优先级 `AboveNormal`（高于 UI/持久化的 Normal，低于采集的 Highest），轻量级阈值比较（< 1ms/帧）。

#### 6. ConsoleApp1 Program 生命周期管理

```csharp
// 启动时：WebAPI 先创建，子进程后连接
coreBus = new CoreDataBus();
coreBus.Open();

// 优雅退出时
coreBus?.Dispose();
```

CoreDataBus 提升为 `Program` 静态单例，由 Program 统一控制启停，Analysis 线程不再自行管理。

#### 7. WebAPI 侧同步

- 注册 `CoreDataBus` 单例
- `ApplicationStarted` 时先创建 CoreDataBus，再启动子进程（确保子进程 `Open()` 时共享内存已存在）
- `CoreDataBus.Create(channels: 2, buffer: 10_000_000, sampleRate: 1_000_000)`
