## 一、背景（Background）

用户在调试系统时发现两个问题了：

1. **配置启动器状态栏不更新**：在 ConfigViewModel 中填写 MQTT 配置并启动 WebAPI 后，底部状态栏的 WebAPI 和 MQTT 连接状态均显示"未连接/—"，尽管 WebAPI 日志显示服务正常运行、MQTT 已连接。
2. **MQTT RPC 未知方法触发断连**：当外部客户端（旧版前端）发送使用旧命名约定 `SYSTEM_STATE`（大写下划线）的 RPC 请求时，服务器发布错误响应到 `{topic}/response`，该主题被自身 `$rpc/{id}/#` 订阅重新捕获，形成 `/response` 无限追加的级联递归，最终 Broker 发出 `DISCONNECT (TopicNameInvalid)` 强断 MQTT 连接。

### 问题（Problem）

#### 1. ConfigLauncher 状态栏（WebAPI/MQTT 连接状态）在 WebAPI 启动后不更新

MainWindowViewModel 的 `AutoDetectWebApi()` 仅在应用启动 200ms 后执行一次连接状态检查——此时 WebAPI 尚未启动（由用户在 Config 标签页手动启动）。当 ConfigViewModel 通过 `StartWebApiAndWait()` 成功启动 WebAPI 后，`onReady` 回调仅切换标签页并调用 `ControlVm.RefreshStateCommand`（更新控制面板状态），但未调用 `MainWindowViewModel.RefreshConnectionStatusAsync()` 更新底部状态栏的 WebAPI/MQTT 连接指示器。

#### 2. MQTT RPC 错误响应主题被服务器自身 /response 递归中毒

`MqttRpcBackgroundService` 订阅了 `$rpc/{MachineId}/#`，`#` 通配符匹配包括 `/response` 在内的所有子主题。当收到未注册的 RPC 方法请求时，代码发布错误响应到 `{requestTopic}/response`。该响应主题又匹配 `#` 订阅，被 `HandleRpcRequestAsync` 重新消费为新的 RPC 请求——每次循环追加一层 `/response`：

```
/response → /response/response → /response/response/response → ...
```

约 7 次追加后主题名超过 Broker 允许长度或格式非法，Broker 发送 `DISCONNECT (ReasonCode=TopicNameInvalid)` 强制断连。

---

## 二、解决方案（Solution）

### 整体思路

1. 在 `onReady` 回调和 `RestartWebApiAsync` 成功分支中各增加一次 `RefreshConnectionStatusAsync()` 调用，确保 WebAPI 启动后就绪时状态栏正确刷新。
2. 在 `HandleRpcRequestAsync` 入口增加主题过滤：若消息主题以 `/response` 结尾，直接返回不处理，切断自循环路径。

### 具体实施

#### 1. ConfigLauncher 状态栏刷新补充

`AvaloniaApplication_ConfigLauncher/ViewModels/MainWindowViewModel.cs`

**onReady 回调**（第 77 行： ConfigViewModel 启动 WebAPI 后就执行）：
```csharp
ControlVm.RefreshStateCommand.Execute(null);
_ = RefreshConnectionStatusAsync();   // ← 新增
```

**RestartWebApiAsync 成功分支**（第 250 行：WebAPI 重启成功后执行）：
```csharp
ControlVm.RefreshStateCommand.Execute(null);
_ = RefreshConnectionStatusAsync();   // ← 新增
```

#### 2. RPC 响应自循环阻断

`WebAPI/Service/MqttRpcBackgroundService.cs`

在 `HandleRpcRequestAsync` 中 `ParseTopic` 成功后、业务逻辑执行前，增加主题过滤守卫：

```csharp
if (!ParseTopic(e.ApplicationMessage.Topic, out var method, out var corrId))
    return;

if (e.ApplicationMessage.Topic.EndsWith("/response", StringComparison.Ordinal))
    return;                                  // ← 新增：阻断自消费

var payload = e.ApplicationMessage.Payload.ToArray();
```

原理：
- 原始 RPC 请求主题格式 `$rpc/{id}/{method}/{corrId}` → 不以 `/response` 结尾 → 正常处理
- 错误响应主题 `$rpc/{id}/{method}/{corrId}/response` → 以 `/response` 结尾 → 直接返回
- `PublishAsync` 的发布行为不受影响（过滤仅作用于入站消息）
