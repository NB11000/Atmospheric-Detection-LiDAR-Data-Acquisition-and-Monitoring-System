## 一、背景（Background）

ConsoleApp1 子进程的 Detection 线程已通过 gRPC 将 `SIGNAL_OBSTRUCTION` 告警上报到 WebAPI，但存在两个问题：

### 问题（Problem）

#### 1. 检测告警使用 Error 消息类型，语义不匹配

Detection 线程调用 `GrpcClient.SendErrorMessage()` 上报信号遮挡，`message_type` 为 `"Error"`。信号遮挡是物理现象产生的告警，不是系统错误。告警数据混入 `daq/{id}/events/state_changed`，与设备断开、采集异常等错误事件无法区分。

#### 2. 结构化上下文丢失

`SendErrorMessage` 将检测数据压成一句自然语言文本（`"通道1信号遮挡 @序号123456"`），StructuredSample 的原始字段（Timestamp、CH1、CH2、Time）全部丢失。前端收到后只能展示字符串，无法做趋势分析或二次加工。

#### 3. WebAPI 缺少检测数据专用服务

WebAPI 没有专门的服务接收结构化检测数据并发布到专属 MQTT topic。需要新增一个与现有 `WaveformPublishService`、`LowFrequencyPublisher`、`PersistenceService` 模式一致的 `IAcquisitionBoundService`。

#### 4. LiDAR 反演管道 issue 状态未更新

5 个 LiDAR 反演相关 issue 已完成实现但状态仍为 `needs-triage`，需同步标记为 `done`。

## 二、解决方案（Solution）

### 整体思路

占位实现检测告警端到端链路：ConsoleApp1 → gRPC `"Detection"` message_type → WebAPI `DetectionPublisherService` → MQTT `daq/{id}/detection/alerts`。不扩展 proto（保持 wire format 稳定），用 JSON 在 `AdResponse.Content` 传输结构化数据。未来用 proto `DetectionAlert` + `Any.Pack` 替换时，路由分支不变。

### 具体实施

#### 1. 新增 `DetectionAlertDto` 数据模型

`WebAPI/Models/DetectionAlertDto.cs`，5 字段 record：AlarmType / Severity / Timestamp / CH1 / CH2。占位阶段双方约定 JSON 字段名对齐，不共享代码。

```csharp
public record DetectionAlertDto
{
    public string AlarmType { get; init; }
    public string Severity { get; init; }
    public long Timestamp { get; init; }
    public double CH1 { get; init; }
    public double CH2 { get; init; }
}
```

#### 2. ConsoleApp1 新增 `GrpcClient.SendDetectionMessage()`

与 `SendErrorMessage` 并列，将告警数据序列化为 JSON 放入 `AdResponse.Content`，`MessageType` 设为 `"Detection"`。

```csharp
GrpcClient.SendDetectionMessage("SIGNAL_OBSTRUCTION", "warning", s.Timestamp, s.CH1, s.CH2);
```

#### 3. DetectionPublisherService — 第 4 个 IAcquisitionBoundService

- 内部 `Channel<DetectionAlertDto>`（与 `DetectionChannel` 同一抽象）
- `OnAlertReceived()` 入队，后台 `ReadAllAsync` 循环消费
- 发布到 `daq/{machineId}/detection/alerts`，QoS 1
- `RequiresMqttConnection = true`，由 `AcquisitionLifecycleCoordinator` 统一管理启停
- MQTT 未连接时静默跳过（参考 `LowFrequencyPublisher` 模式）

#### 4. GrpcServiceImpl 新增 Detection 消息路由

构造函数注入 `DetectionPublisherService`，在 `Communicate()` 循环中新增 `"Detection"` 分支：JSON 反序列化 → `OnAlertReceived()`。

#### 5. 5 个 xUnit 测试

`RequiresMqttConnection` 返回 true、Start/Stop 幂等、Stop 无 Start 不抛异常、端到端处理循环。

#### 6. CONTEXT.md 术语更新

新增"检测告警 (Detection Alert)"术语、更新 Relationships 中检测线程 → MQTT 的完整数据流链路、将"检测发布服务"纳入 MQTT 绑定服务列表。

## 五、架构影响

| 维度 | 变更前 | 变更后 |
|------|--------|--------|
| 告警传输 | Error message_type + 自然语言文本 | Detection message_type + JSON 结构化数据 |
| MQTT topic | 混入 `events/state_changed` | 专属 `detection/alerts` |
| IAcquisitionBoundService | 3 个（波形/低频/持久化） | 4 个（+检测发布） |
| 生命周期管理 | Coordinator 管理 3 个服务 | Coordinator 管理 4 个服务，无额外集成 |

**数据链路**：
```
Detection 线程 → SendDetectionMessage(JSON) → gRPC "Detection"
  → GrpcServiceImpl.Deserialize<DetectionAlertDto>
  → DetectionPublisherService.OnAlertReceived()
  → Channel<DetectionAlertDto> → RunLoopAsync
  → MQTT daq/{id}/detection/alerts (QoS 1)
```
