# WebSocket端到端测试指南

**创建时间：** 2026-04-08  
**适用版本：** 数据采集与检测系统 V2.0  
**测试目标：** 验证完整的WebSocket数据流，从数据采集到UI展示

## 测试概述

本指南详细说明如何测试WebSocket数据流的完整端到端功能，包括：
1. 服务器启动与WebSocket端点可用性
2. 客户端连接与数据接收
3. UI图表实时更新
4. 错误处理与重连机制

## 测试环境准备

### 1. 软件环境
- .NET 8 SDK 已安装
- 数据采集系统源代码完整
- 采集卡硬件（可选，可使用模拟数据）

### 2. 网络环境
- 本地测试：localhost (127.0.0.1)
- 跨机器测试：确保网络互通，防火墙开放5135端口

### 3. 测试工具
- 已提供的 `WebSocket测试客户端` (控制台程序)
- 浏览器开发者工具 (用于WebSocket调试)
- 网络监控工具 (如Wireshark，可选)

## 测试步骤

### 第一阶段：基础连接测试

#### 步骤1.1：启动WebAPI服务器
```bash
# 打开命令行，进入WebAPI目录
cd "数据采集与检测系统V2.0\WebAPI"

# 启动服务器
dotnet run

# 预期输出：
# 环境: Development
# 当前环境: 开发环境
# 本机局域网地址：
# 👉 192.168.1.22:5135
# Now listening on: http://[::]:5135
# 子进程已启动...
```

#### 步骤1.2：验证WebSocket端点
使用浏览器开发者工具或curl测试：
```bash
# 使用curl测试HTTP端点（WebSocket需要特殊工具）
curl http://localhost:5135/ws/ui-data

# 预期响应：400 Bad Request，提示"此端点仅支持WebSocket连接"
```

#### 步骤1.3：运行测试客户端
```bash
# 新开命令行，进入测试客户端目录
cd "数据采集与检测系统V2.0\开发记录\WebSocket测试客户端"

# 运行测试客户端
dotnet run

# 预期输出：
# ✅ 连接成功！
# 连接状态: Open
# 📊 开始接收数据帧...
# （此时无数据，因为尚未开始采集）
```

### 第二阶段：数据流测试

#### 步骤2.1：启动数据采集
1. 运行Avalonia客户端应用程序
   ```bash
   cd "数据采集与检测系统V2.0\AvaloniaApplication1"
   dotnet run
   ```

2. 在Avalonia界面中：
   - 点击"打开采集卡"按钮
   - 点击"开始采集"按钮
   - 观察状态显示："WebSocket数据流已连接"

#### 步骤2.2：验证数据流
在测试客户端窗口应看到：
```
📊 统计: 30帧, 30.0FPS, 3.84Mbps
📈 样本值 - 通道1[0]: 1.234567, 通道2[0]: 2.345678
```

#### 步骤2.3：验证UI更新
在Avalonia客户端中：
- 观察Chart1图表是否实时更新
- 检查数据点是否与测试客户端显示的值匹配
- 验证两个通道的数据都正确显示

### 第三阶段：功能完整性测试

#### 步骤3.1：停止/恢复采集
1. 点击"停止采集"按钮
2. 观察测试客户端：数据流停止
3. 再次点击"开始采集"按钮
4. 验证数据流恢复

#### 步骤3.2：跨机器测试（可选）
1. 在服务器机器上运行WebAPI
2. 在客户端机器上运行Avalonia应用程序
3. 修改客户端配置中的服务器地址
4. 验证跨机器数据流

#### 步骤3.3：多客户端测试
1. 同时运行多个测试客户端实例
2. 验证所有客户端都能接收数据
3. 观察服务器性能表现

### 第四阶段：错误处理测试

#### 步骤4.1：网络中断测试
1. 开始正常数据流
2. 断开网络连接（禁用网卡或拔网线）
3. 观察客户端重连行为
4. 恢复网络连接
5. 验证自动重连成功

#### 步骤4.2：服务器重启测试
1. 开始正常数据流
2. 停止WebAPI服务器 (Ctrl+C)
3. 观察客户端错误处理
4. 重启WebAPI服务器
5. 验证客户端自动重连

#### 步骤4.3：异常数据测试
1. 修改服务器发送异常数据（如错误帧大小）
2. 观察客户端错误处理
3. 验证客户端不会崩溃

## 测试验收标准

### 必须通过
- [ ] 客户端能成功连接到WebSocket端点
- [ ] 数据采集开始后能收到数据帧
- [ ] 数据帧大小正确（16,000字节）
- [ ] UI图表能实时更新
- [ ] 停止采集后数据流停止
- [ ] 网络中断后能自动重连
- [ ] 服务器重启后能恢复连接
- [ ] 无内存泄漏（长时间运行测试）

### 期望通过
- [ ] 跨机器数据传输正常
- [ ] 多客户端并发连接稳定
- [ ] 帧率稳定在30±2 FPS
- [ ] 端到端延迟 < 50ms
- [ ] CPU使用率 < 30% (客户端)
- [ ] 内存使用稳定无增长

## 性能基准测试

### 测试方法
1. 运行数据流至少5分钟
2. 记录以下指标：
   - 平均帧率
   - 最大/最小帧间隔
   - 数据丢失率
   - 客户端CPU/内存使用
   - 网络带宽使用

### 合格标准
- **帧率稳定性**: 30±2 FPS (90%的时间)
- **数据完整性**: 丢包率 < 0.1%
- **延迟**: 端到端 < 50ms
- **资源使用**: 客户端内存 < 200MB，CPU < 30%

## 常见问题排查

### 问题1：无法连接WebSocket
**症状**: 测试客户端显示连接错误
**排查步骤**:
1. 确认WebAPI服务器正在运行
   ```bash
   netstat -ano | findstr :5135
   ```
2. 检查防火墙设置
3. 验证URL格式：`ws://localhost:5135/ws/ui-data`
4. 检查服务器日志中的错误信息

### 问题2：连接成功但无数据
**症状**: 连接成功，但无数据帧接收
**排查步骤**:
1. 确认数据采集已开始（Avalonia界面显示"正在采集..."）
2. 检查子进程(ConsoleApp1)是否正常运行
3. 验证共享内存是否正常写入
4. 检查WebAPI日志中是否有WebSocket发送记录

### 问题3：数据帧大小不正确
**症状**: 收到异常帧大小警告
**排查步骤**:
1. 确认服务器和客户端使用相同的数据格式
2. 检查`HandleWebSocketConnection`方法中的缓冲区大小
3. 验证共享内存读取逻辑

### 问题4：UI更新延迟高
**症状**: 数据接收正常，但UI更新卡顿
**排查步骤**:
1. 检查UI线程是否被阻塞
2. 验证`Dispatcher.UIThread.Post`使用正确
3. 检查图表渲染性能
4. 减少非必要的UI操作
