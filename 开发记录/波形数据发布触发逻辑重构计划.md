# 波形数据发布触发逻辑重构计划

## 一、可行性分析

### 用户反馈的原始问题

原方案存在两个架构隐患：

1. **跨层复杂度**：触发链 `CollectorHandler → GrpcServiceImpl → SystemStateService → MqttRpcBackgroundService` 跨越 4 层，流程过于曲折
2. **发布机制割裂**：`MqttRpcBackgroundService` 内置波形发布循环，而 `MqttEventPublisher` 作为独立类承担 state/data/alarm 事件发布，功能重复且职责不清

### "普通数据发布模式"可行性判断

| 维度 | 评估 | 说明 |
|------|------|------|
| 统一发布通道 | ✅ 可行 | 新建 `WaveformPublishAsync` 方法到 `MqttEventPublisher`，波形发布与 state/data/alarm 事件共用同一 MQTT 客户端 |
| 简化触发链 | ✅ 可行 | 触发链缩短为 2 跳：`SystemStateService.AcquiringStateChanged → WaveformPublishService` |
| 消除轮询 | ⚠️ 不可行 | 共享内存是拉模型（Pull），WebAPI 侧必须轮询读取。除非改造子进程直接推 MQTT（改动过大，不推荐）。轮询是本质上不可避免的，但可以把轮询逻辑解耦到一个独立、轻量的服务中 |
| 职责分离 | ✅ 可行 | `MqttRpcBackgroundService` 退化为纯 RPC 路由服务；新建 `WaveformPublishService` 专注波形发布 |

### 当前冗余分析

```
WebSocket 循环 (Tool.cs:90-126)    —— 每 33ms 读共享内存 → 推送 WebSocket
MQTT 波形循环 (MqttRpc:370-414)   —— 每 Nms 读共享内存 → 推送 MQTT 波形主题
                                    ↑ 同一个数据源被两处独立轮询
```

结论：轮询无法避免，但可以把轮询发布逻辑从 RPC 服务中剥离，封装为独立的轻量服务。

## 二、重构后架构

```
┌─────────────────────────────────────────────────┐
│ MqttRpcBackgroundService (纯 RPC)               │
│  ─ 连接/断连/重连管理                            │
│  ─ RPC 请求路由分发                              │
│  ─ 遗嘱消息                                      │
│  ─ 设置 MqttEventPublisher.MqttClient           │
└─────────────────────────────────────────────────┘
                        │ 共享 MqttEventPublisher.MqttClient
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ State 事件   │ │ Alarm 事件   │ │ Waveform 发布│
│ (低频 JSON)  │ │ (低频 JSON)  │ │ (高频 Binary)│
│ MqttEvent    │ │ MqttEvent    │ │ Waveform     │
│ Publisher    │ │ Publisher    │ │ Publish      │
│              │ │              │ │ Service      │
└──────────────┘ └──────────────┘ └──────────────┘
                        均通过 MqttEventPublisher.MqttClient.PublishAsync
```

**触发链（仅 2 跳）：**

```
采集开始：
  GrpcServiceImpl.UpdateStateFromCommandResponse("START_AD")
    → SystemStateService.UpdateCollectorState(Acquiring=true)
      → AcquiringStateChanged?.Invoke(true)          ← 第 1 跳
        → WaveformPublishService.Start()             ← 第 2 跳
          → 启动 PeriodicTimer → 读共享内存 → MqttEventPublisher.PublishWaveformAsync

采集停止：
  (STOP_AD / CLOSE_DEVICE / DEVICE_DISCONNECTED / 进程断开)
    → SystemStateService.UpdateCollectorState(Acquiring=false)
      → AcquiringStateChanged?.Invoke(false)
        → WaveformPublishService.Stop()
```

## 三、具体改动文件与步骤

### Step 1：`MqttEventPublisher.cs` — 新增波形发布方法

**目的**：将 MQTT 波形发布能力纳入统一的 `MqttEventPublisher`，使波形发布也走"普通数据发布"通道。

**改动**：

- 新增 `PublishWaveformDataAsync(byte[] ch1Payload, byte[] ch2Payload)` 方法
- 使用 `MqttClient`（由 `MqttRpcBackgroundService` 连接时注入）发布到 `daq/{machineId}/waveform/ch1` 和 `ch2`
- QOS 0（至多一次），与现有波形发布策略一致
- 包含连接检查和异常捕获

### Step 2：`WebAPI/Service/WaveformPublishService.cs` — 新建波形发布后台服务

**目的**：将波形发布逻辑从 `MqttRpcBackgroundService` 剥离，成为一个独立、轻量的 `BackgroundService`。

**内容**：

- 类名：`WaveformPublishService`，继承 `BackgroundService`
- 核心成员：
  - `_mqttEventPublisher`：注入，用于调用统一的发布方法
  - `_systemStateService`：注入，用于读取采集状态
  - `_uiSharedBuffer`：注入，用于读取共享内存波形帧
  - `_waveformBuf1 / _waveformBuf2`：缓冲区（double[]）
  - `_ch1Bytes / _ch2Bytes`：字节缓冲区（从 ArrayPool 租用）
  - `_waveformLock`、`_isRunning`：控制启停的线程安全标志
  - `_waveformCts`：取消令牌
- 核心方法：
  - `ExecuteAsync(CancellationToken)`：空实现（由事件驱动，不在此处启动循环）
  - `Start()`：线程安全的启动方法
  - `Stop()`：线程安全的停止方法
  - `RunLoopAsync(CancellationToken)`：内部循环 — PeriodicTimer → 读共享内存 → `_mqttEventPublisher.PublishWaveformDataAsync`
- 事件订阅：
  - 订阅 `SystemStateService.AcquiringStateChanged`
  - `true` → `Start()`；`false` → `Stop()`
- 重连感知：
  - MQTT 重连后，若采集状态为 `Acquiring=true`，需重启循环。可通过在 `MqttRpcBackgroundService.ConnectAsync` 成功后主动调用 `WaveformPublishService.Start()` 实现（如果当前正在采集且未运行）。

### Step 3：`SystemStateService.cs` — 添加 `AcquiringStateChanged` 事件

**目的**：提供采集状态变更通知，作为波形发布启停的唯一触发源。

**改动**：

- 新增 `public event Action<bool>? AcquiringStateChanged;`
- 在 `UpdateCollectorState` 中比较新旧 `Acquiring` 值，仅变化时触发事件

> 注：此步骤与原计划一致，已部分实现（编辑文件已回退需重新应用）。

### Step 4：`MqttRpcBackgroundService.cs` — 删除波形发布逻辑

**目的**：移除非 RPC 职责，做到单一职责。

**改动**：

- **删除** `WaveformFramePoints`、`WaveformFrameBytes` 常量（移至 `WaveformPublishService`）
- **删除** `_waveformBuf1`、`_waveformBuf2` 字段
- **删除** `_ch1Bytes`、`_ch2Bytes` 字段（ArrayPool 归还逻辑移到 `WaveformPublishService.Dispose`）
- **删除** `_ch1Template`、`_ch2Template` 字段
- **删除** `_waveformCts` 字段（移至 `WaveformPublishService`）
- **删除** `_waveformCh1Topic`、`_waveformCh2Topic` 变量（移至 `MqttEventPublisher` 或 `WaveformPublishService`）
- **删除** `PublishWaveformLoopAsync` 方法（整体迁移到 `WaveformPublishService`）
- **删除** `ConnectAsync` 中波形循环启动代码（第 227-248 行，模板构建和循环启动）
- **删除** `OnDisconnectedAsync` 中 `_waveformCts?.Cancel()`（改为通知 `WaveformPublishService.Stop()`）
- **删除** `StopAsync` 中 `_waveformCts?.Cancel()`
- **删除** `Dispose` 中 `_waveformCts?.Dispose()` 和 ArrayPool 归还（移至 `WaveformPublishService.Dispose`）
- **简化** `ConnectAsync` 末尾：连接成功后，不再启动波形循环
- **新增** `ConnectAsync` 末尾：如果当前采集状态为 `Acquiring=true`，通知 `WaveformPublishService.Start()` 恢复波形发布

### Step 5：`Program.cs` — 注册新服务

**改动**：

- 添加 `builder.Services.AddSingleton<WaveformPublishService>();`
- 添加 `builder.Services.AddHostedService<WaveformPublishService>(sp => sp.GetRequiredService<WaveformPublishService>());`

## 四、触发条件汇总

| 触发场景 | Acquiring 变化 | 触发链 | 结果 |
|----------|---------------|--------|------|
| 用户调用 `collector-start-ad` RPC → 子进程返回 `AD_STARTED` | false→true | `AcquiringStateChanged(true)` → `WaveformPublishService.Start()` | 波形循环启动 |
| 用户调用 `collector-stop-ad` RPC | true→false | `AcquiringStateChanged(false)` → `WaveformPublishService.Stop()` | 波形循环停止 |
| 用户调用 `collector-close-device` RPC | true→false | 同上 | 同上 |
| 子进程断开（gRPC 断连） | true→false | `ResetCollectorState()` → `UpdateCollectorState(Acquiring=false)` → 事件触发 | 同上 |
| 子进程上报 `DEVICE_DISCONNECTED` 错误 | true→false | `UpdateStateFromError` → `UpdateCollectorState(Acquiring=false)` → 事件触发 | 同上 |
| 子进程上报 `ACQUISITION_FAILED` 错误 | true→false | 同上 | 同上 |
| MQTT 断连（采集仍运行中） | 不变 | `WaveformPublishService` 内部检测到 MQTT 未连接 → 跳过本次发布 | 循环继续但不发布 |
| MQTT 重连成功（采集仍运行中） | 不变 | `MqttRpcBackgroundService.ConnectAsync` 末尾主动调用 `WaveformPublishService.Start()` | 波形发布恢复 |
| WebAPI 进程优雅停止 | true→false | `StopAsync` → `ResetCollectorState` → 事件触发 → `WaveformPublishService.Stop()` | 波形循环停止 |

## 五、同步与异常处理

| 场景 | 机制 |
|------|------|
| 并发 Start/Stop 调用 | `lock (_waveformLock)` + `_isRunning` 标记，防止重复启停 |
| 已在运行时收到 Start | `Start()` 内检查 `_isRunning`，直接返回 |
| 已停止时收到 Stop | `Stop()` 内检查 `!_isRunning`，直接返回 |
| 循环内部发布失败 | try-catch 捕获，记录日志，下次循环自动重试 |
| MQTT 未连接时 Start | `RunLoopAsync` 内每轮检查 `MqttClient?.IsConnected`，未连接则跳过 |
| 共享内存数据不足 | `ReadLatestFrame` 返回归零数据（已有逻辑），不会崩溃 |
| ArrayPool 泄漏 | `Dispose` 中统一归还 `_ch1Bytes`、`_ch2Bytes` |
| CTS 泄漏 | `Dispose` 中统一释放 `_waveformCts` |

## 六、前瞻性分析：多数据类型发布与采集状态关联

### 问题 1：波形数据与其他数据能否整合在同一服务中？

**数据发布的触发模式分类：**

| 模式 | 示例 | 触发方式 | 频率 | 载荷格式 |
|------|------|---------|------|---------|
| 事件驱动型 | StateChanged、DeviceAlarm | 外部类调用（被动） | 低（状态切换时） | JSON |
| 定时轮询型 | Waveform（共享内存） | 内部 PeriodicTimer（主动） | 高（每 Nms） | Binary |
| 流驱动型 | 未来可能的数据管道 | Channel/数据到达回调（被动） | 中（不定） | 不定 |

**整合可行性评估：**

- 事件驱动型和定时轮询型的**触发机制根本不同**。前者是"被调用"，后者是"主动跑"。强行塞入同一个类会导致该类同时扮演两种角色。
- 但如果未来有多个定时轮询型数据（波形 + 其他共享内存数据），它们读同一个数据源、用同一个计时器、走同一套启停逻辑———**这些应该整合**。

**推荐设计：**

`WaveformPublishService` 保持独立，但在设计上预留扩展点：
- 类名不改为 `DataPublishingService`（避免过早抽象），但内部方法用通用命名
- `RunLoopAsync` 中每次 tick 的执行体设计为可扩展的管道：读共享内存 → 转换 → 发布，未来添加新数据类型只需加一个转换步骤
- 所有定时轮询型数据的发布出口统一走 `MqttEventPublisher.PublishXxxAsync`

```
WaveformPublishService (定时轮询触发器)
  │
  ├── Tick 1: 读共享内存波形帧 → MqttEventPublisher.PublishWaveformDataAsync
  │
  └── (未来扩展) Tick 2: 读共享内存其他数据 → MqttEventPublisher.PublishTelemetryAsync
```

而事件驱动的发布（StateChanged、Alarm）则由各自的业务类直接调用 `MqttEventPublisher`，与定时轮询完全解耦。

```
GrpcServiceImpl ──→ MqttEventPublisher.PublishStateChangedAsync   (事件驱动)
CniLaser         ──→ MqttEventPublisher.PublishStateChangedAsync   (事件驱动)
```

### 问题 2：数据发布流程是否必须与采集状态关联？

**结论：需要保留，但按数据类型区分。**

| 数据类型 | 需要采集门控？ | 理由 |
|---------|--------------|------|
| 波形数据 (16KB/帧) | **是** | 无采集时共享内存为归零值，发布无意义且浪费带宽。订阅方也需知道何时数据有效 |
| 系统健康指标 | 否 | 应与采集状态无关，始终上报 |
| 错误/告警 | 否 | 错误不应因采集停止而抑制 |
| 未来遥测数据 | 按需决定 | 由具体数据类型自行判断 |

**设计原则：门控机制由各数据发布者按需声明，而非全局强制。**

当前计划的实现方式已经符合这个原则：
- `SystemStateService.AcquiringStateChanged` 是一个**通知事件**，不强制任何行为
- `WaveformPublishService` **选择订阅**该事件作为自己的启停条件
- 未来的其他发布服务可以**选择不订阅**，也可以订阅其他事件（如 `DeviceOpened`）
- `MqttEventPublisher` 内部的 `PublishStateChangedAsync` 等事件驱动方法，本身就不受 `Acquiring` 门控

```csharp
// WaveformPublishService 选择性地使用门控
stateService.AcquiringStateChanged += (isAcquiring) =>
{
    if (isAcquiring) Start(); else Stop();
};

// 未来某个始终运行的数据发布服务 —— 不订阅此事件即可
// 或者订阅但忽略（始终运行）
```

## 七、新旧方案对比

| 维度 | 原方案（已否决） | 新方案 |
|------|-----------------|--------|
| 触发链跳数 | 4-5 跳（CollectorHandler → GrpcServiceImpl → SystemStateService → MqttRpcBackgroundService） | 2 跳（SystemStateService → WaveformPublishService） |
| 波形发布归属 | 嵌入 MqttRpcBackgroundService（职责混合） | 独立 WaveformPublishService（单一职责） |
| MQTT 发布出口 | 波形单独构建 MqttApplicationMessage | 统一走 MqttEventPublisher |
| 多数据类型扩展 | 无考虑 | `RunLoopAsync` 设计为可扩展管道；门控机制按数据类型区分而非全局强制 |
| MqttRpcBackgroundService 改动 | 约 60 行新增 | 约 80 行删除（大幅简化） |
| 新增代码量 | 少 | 约 120 行（新文件 WaveformPublishService.cs） |
| 可测试性 | 差（强耦合） | 好（可独立单元测试波形启停逻辑） |
