# 跨平台进程生命周期监控系统优化方案

## 1. 背景与问题分析

### 1.1 原系统存在的问题

在现有的WebAPI数据采集系统中，主进程（WebAPI）通过gRPC与数据采集子进程（ConsoleApp1）通信。系统的优雅关闭机制存在以下缺陷：

1. **信号处理不完整**：
   - 仅能处理Ctrl+C（SIGINT）信号
   - 直接关闭控制台窗口（Windows发送`CTRL_CLOSE_EVENT`）时无法触发优雅关闭
   - 进程被强制终止、系统关机等情况完全无法处理

2. **单向依赖问题**：
   - 子进程完全依赖主进程通过gRPC发送`EXIT`命令来触发退出
   - 主进程异常退出时，子进程无法感知，导致资源泄漏和设备状态异常

3. **设备状态风险**：
   - 设备正在采集时直接退出可能导致硬件资源未释放
   - 可能造成设备控制混乱，影响后续使用

### 1.2 需求目标

建立健壮的跨平台进程生命周期监控系统，确保：

1. **全面覆盖**：支持主进程正常退出（Ctrl+C）和异常退出（直接关闭窗口、强制终止等）
2. **事件驱动**：使用异步等待机制，避免CPU轮询开销
3. **跨平台兼容**：不使用Windows专用API，支持Windows/Linux/macOS
4. **设备安全**：退出时根据设备状态执行安全清理流程
5. **向后兼容**：保持现有gRPC通信机制，不影响正常使用

## 2. 解决方案设计

### 2.1 核心设计思想

采用**进程句柄继承 + WaitForExitAsync**的事件驱动监控模式：

```
┌─────────────────────────────────────────────────────────────┐
│                     双重退出保障体系                         │
├─────────────────────────────────────────────────────────────┤
│ 路径1：gRPC命令退出（正常流程）                            │
│  ├─ 主进程通过gRPC发送EXIT命令                              │
│  ├─ 子进程响应EXIT_OK                                       │
│  └─ 执行设备清理，停止gRPC连接                              │
│                                                            │
│ 路径2：父进程监控退出（异常流程）                          │
│  ├─ 子进程通过WaitForExitAsync监控父进程                     │
│  ├─ 检测到父进程退出                                        │
│  ├─ 触发独立优雅退出流程                                    │
│  └─ 不发送gRPC响应（主进程已关闭）                          │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 技术选型

- **监控机制**：`Process.WaitForExitAsync()`（.NET Core 3.0+）
- **事件模型**：异步等待，零CPU占用
- **平台兼容**：标准.NET API，跨平台支持
- **参数传递**：命令行参数传递父进程ID

```csharp
// 文件：ConsoleApp1/Program.cs

// 新增字段
private static CancellationTokenSource _parentMonitorCts;
private static GrpcClient _grpcClient;

// 参数解析增强
int parentProcessId = -1;
if (args.Length >= 3 && int.TryParse(args[2], out int parsedPid))
{
    parentProcessId = parsedPid;
    logger.LogInformation($"父进程ID：{parentProcessId}，启动父进程监视");
}

// 启动父进程监控
if (parentProcessId > 0)
{
    _parentMonitorCts = new CancellationTokenSource();
    _ = Task.Run(() => MonitorParentProcessAsync(parentProcessId, _parentMonitorCts.Token));
}
```

#### 3.2.2 MonitorParentProcessAsync - 事件驱动监控

```csharp
private static async Task MonitorParentProcessAsync(int parentProcessId, CancellationToken cancellationToken)
{
    try
    {
        // 获取父进程对象
        Process parentProcess = Process.GetProcessById(parentProcessId);
        
        // 使用WaitForExitAsync异步等待父进程退出
        await parentProcess.WaitForExitAsync(cancellationToken);
        
        if (!cancellationToken.IsCancellationRequested)
        {
            // 检测到父进程退出，触发优雅退出
            await TriggerGracefulExitFromParentMonitorAsync();
        }
        
        parentProcess?.Dispose();
    }
    catch (ArgumentException)
    {
        // 父进程不存在，直接触发退出
        await TriggerGracefulExitFromParentMonitorAsync();
    }
    // ... 异常处理 ...
}
```

#### 3.2.3 TriggerGracefulExitFromParentMonitorAsync - 独立退出流程

```csharp
private static async Task TriggerGracefulExitFromParentMonitorAsync()
{
    try
    {
        logger.LogInformation("开始父进程监控触发的优雅退出流程...");
        
        // 1. 停止数据采集（如果设备正在采集）
        if (aD_Controlcs != null)
        {
            aD_Controlcs.stop(); // 停止采集
            logger.LogInformation("数据采集已停止");
            
            // 2. 关闭采集卡设备
            if (aD_Controlcs.mHandle >= 0)
            {
                USB1602.USB1602_CloseDevice(aD_Controlcs.mHandle);
                aD_Controlcs.mHandle = -1;
                logger.LogInformation("采集卡设备已关闭");
            }
        }
        
        // 3. 释放共享内存资源
        if (uISharedBuffer != null)
        {
            uISharedBuffer.Dispose();
            logger.LogInformation("共享内存资源已释放");
        }
        
        // 4. 停止gRPC客户端（不发送响应）
        if (_grpcClient != null)
        {
            _grpcClient.Stop();
        }
        
        logger.LogInformation("父进程监控触发的优雅退出流程完成");
    }
    catch (Exception ex)
    {
        logger.LogError($"优雅退出过程中发生异常：{ex.Message}");
    }
    finally
    {
        // 确保进程退出
        Environment.Exit(0);
    }
}
```

## 4. 工作原理

### 4.1 正常退出流程（gRPC命令）

```
┌─────────┐     EXIT命令    ┌─────────┐
│ 主进程  │ ──────────────> │ 子进程  │
│ (WebAPI)│                 │(Console)│
└─────────┘                 └─────────┘
                                │
                                │ 1. 响应EXIT_OK
                                │ 2. 停止采集
                                │ 3. 关闭设备
                                │ 4. 停止gRPC连接
                                │ 5. 退出进程
```

### 4.2 异常退出流程（父进程监控）

```
┌─────────┐                 ┌─────────┐
│ 主进程  │                 │ 子进程  │
│ (WebAPI)│                 │(Console)│
└─────────┘                 └─────────┘
     │                           │
     │ 异常退出                 │ 1. WaitForExitAsync触发
     ▼                           │ 2. 检测到父进程退出
   [退出]                       │ 3. 触发独立优雅退出
                                │ 4. 停止采集，关闭设备
                                │ 5. 释放资源
                                │ 6. 退出进程（不发送响应）
                                ▼
                              [退出]
```

### 4.3 并发处理机制

- **监控任务与gRPC任务并行运行**
- **取消令牌协调**：gRPC正常退出时取消监控任务
- **资源清理幂等性**：设备控制方法确保多次调用安全

## 6. 优势与特点

### 6.1 技术优势

1. **事件驱动，零轮询**：
   - 使用`WaitForExitAsync()`实现真正的异步等待
   - 无CPU占用，资源效率高

2. **跨平台兼容**：
   - 纯.NET标准API实现
   - 支持Windows/Linux/macOS
   - 无平台专用代码

3. **双重保障机制**：
   - gRPC命令退出（正常流程）
   - 父进程监控退出（异常流程）
   - 覆盖所有退出场景

4. **设备安全优先**：
   - 退出前确保停止采集
   - 设备资源完全释放
   - 避免硬件状态异常

### 6.2 工程优势

1. **向后兼容**：
   - 父进程ID为可选参数
   - 旧版本主进程可继续使用
   - 逐步升级，无破坏性变更

2. **代码可维护性**：
   - 监控逻辑集中独立
   - 清晰的退出路径分离
   - 完善的错误处理和日志

3. **扩展性**：
   - 监控机制可复用其他场景
   - 设备状态感知可进一步扩展
   - 支持多种异常检测策略

## 7. 注意事项

### 7.1 部署要求

1. **.NET版本**：需要.NET Core 3.0或更高版本（支持`WaitForExitAsync`）
2. **进程权限**：子进程需要权限访问父进程信息
3. **平台差异**：不同平台的进程管理行为可能有细微差异

### 7.2 安全考虑

1. **进程欺骗防护**：父进程ID来自可信源（主进程传递）
2. **资源清理保障**：确保finally块中执行关键清理
3. **超时处理**：监控任务应有适当的超时机制

### 7.3 性能影响

1. **内存开销**：额外维护Process对象引用
2. **启动延迟**：监控任务启动时间可忽略不计
3. **运行时开销**：异步等待无CPU占用
