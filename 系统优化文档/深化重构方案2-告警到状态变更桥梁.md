## 一、问题背景

### 1.1 现状：告警与状态变更是两条完全断开的路径

根据系统的"强乐观 UI + 命令链路正确性"架构约定：

- **路径 [A]（命令链路）**：MQTT RPC 命令成功 → 更新缓存，不广播
- **路径 [B]（非命令链路）**：异常事件 → 更新缓存，必须广播 `state_changed`

**检测告警属于路径 [B]**：它不由 MQTT RPC 命令触发，没有命令响应作为确认载体。但当前告警仅发布到 `detection/alerts` 主题，**从不触发 `events/state_changed`**。

```
检测告警路径：
  子进程 Detection 线程 → gRPC "Detection" 消息
  → GrpcServiceImpl → DetectionPublisherService
  → MQTT daq/{id}/detection/alerts

状态变更路径：
  子进程 command_response / Error 消息
  → GrpcServiceImpl → SystemStateService.UpdateCollectorState()
  → (手动调用) MqttEventPublisher.PublishStateChangedAsync()
  → MQTT daq/{id}/events/state_changed
```

这两个路径之间**没有任何交汇点**。当检测告警发生时（例如 CH1 信号遮断），`detection/alerts` 主题有消息，但 `events/state_changed` 主题无任何事件。

### 1.2 影响

远端监控系统的订阅方需要：
1. 订阅 `events/state_changed` 获取设备运行状态
2. 订阅 `detection/alerts` 获取检测告警
3. 在两个主题之间自行做时间线关联，推断设备真实健康状态

这违背了"一个概念一个主题"的设计原则。设备健康状态被碎片化到了两个 MQTT 主题中。

### 1.3 检测告警的类型

当前 `DetectionPublisherService` 处理的告警结构（`DetectionAlertDto`）：

```csharp
public class DetectionAlertDto
{
    public string AlarmType { get; set; }   // 告警类型
    public int Severity { get; set; }        // 严重程度 1-5
    public DateTime Timestamp { get; set; }
    public double CH1 { get; set; }
    public double CH2 { get; set; }
}
```

子进程 Detection 线程在 `AD_Controlcs.cs` 中检测 CH1 信号遮挡（CH1 < 0.01V 持续），通过 gRPC 上报。

## 二、优化目标

在 `DetectionPublisherService` 和 `SystemStateService` 之间建立桥梁，使检测告警能够触发设备状态变更，让 `events/state_changed` 主题反映设备的完整健康状态。

核心原则：
- **告警即状态**：当告警代表设备进入降级状态时，自动触发状态变更
- **单一订阅**：远端监控只需订阅 `events/state_changed` 即可获知设备整体健康状态
- **告警主题保留**：`detection/alerts` 仍保留原始告警流，供需要详细告警信息的消费者使用

---

## 三、方案设计

### 3.1 核心思路

在 `DetectionPublisherService.OnAlertReceived()` 中，根据告警类型和严重程度判断是否需要触发状态变更。同时扩展 `SystemStateDto`，增加设备告警状态的字段。

### 3.2 架构图

```
子进程 Detection 线程
        │
        │ gRPC "Detection" 消息
        ▼
  GrpcServiceImpl
        │
        │ OnAlertReceived(alert)
        ▼
  DetectionPublisherService
        │
        ├──→ MQTT daq/{id}/detection/alerts  (保留：原始告警流)
        │
        └──→ SystemStateService.UpdateAlarmState(alert)
                │
                ├── 更新 AlarmState 缓存
                ├── 比较新旧告警状态差异
                └──→ MQTT daq/{id}/events/state_changed  (新增：告警触发状态变更)
```

#### 新增 AlarmStateDto

```csharp
/// <summary>
/// 设备告警状态快照
/// </summary>
public class AlarmStateDto
{
    /// <summary>
    /// 当前活跃告警的最高严重度（0=无告警，1-5=告警等级）
    /// </summary>
    public int ActiveSeverity { get; set; }

    /// <summary>
    /// 当前活跃告警数量
    /// </summary>
    public int ActiveAlarmCount { get; set; }

    /// <summary>
    /// 最近告警的时间戳
    /// </summary>
    public DateTime LastAlarmTime { get; set; }

    /// <summary>
    /// 最近告警的类型列表（最多保留 5 条）
    /// </summary>
    public List<string> RecentAlarmTypes { get; set; } = new();

    /// <summary>
    /// 设备是否处于告警状态
    /// </summary>
    public bool IsAlarming => ActiveSeverity > 0;
}
```

### 3.4 告警 → 状态变更映射规则

| 告警类型 | 严重度阈值 | 触发的状态事件 | 说明 |
|---|---|---|---|
| CH1 信号遮挡 | ≥ 3 | `device_degraded` | 采集质量下降 |
| CH2 信号遮挡 | ≥ 3 | `device_degraded` | 采集质量下降 |
| 设备异常 | ≥ 2 | `device_error` | 硬件级异常 |
| 任意告警 | ≥ 4 | `device_critical` | 严重告警 |
| 告警恢复 | 0 | `device_normal` | 严重度归零=恢复正常 |

### 3.5 告警恢复机制

告警状态不会自动清除。需要引入"告警恢复"逻辑：

- **主动清除**：通过 MQTT RPC `system-clear-alarm` 手动清除
- **超时清除**：告警产生后，如果在配置时间内（默认 60 秒）未收到新的同类型告警，自动降级
- **恢复上报**：子进程 Detection 线程可在检测到信号恢复时，发送 `Severity=0` 的告警消息

### 3.6 SystemStateService 新增方法

```csharp
public class SystemStateService
{
    // === 新增：告警状态管理 ===

    /// <summary>
    /// 更新告警状态（由 DetectionPublisherService 调用）
    /// 当告警严重度变化时，自动触发状态变更事件发布
    /// </summary>
    public void UpdateAlarmState(DetectionAlertDto alert)
    {
        var current = _cachedAlarmState;
        var newState = EvaluateAlarmState(current, alert);

        if (newState.ActiveSeverity != current.ActiveSeverity)
        {
            // 严重度变化 → 触发状态变更事件
            string eventType = newState.ActiveSeverity == 0
                ? "device_normal"
                : newState.ActiveSeverity >= 4
                    ? "device_critical"
                    : "device_degraded";

            PublishStateChangedInternal(eventType, "alarm",
                $"告警状态变更: {alert.AlarmType}",
                $"严重度: {current.ActiveSeverity} → {newState.ActiveSeverity}");
        }

        _cachedAlarmState = newState;
    }

    /// <summary>
    /// 清除所有活跃告警
    /// </summary>
    public void ClearAlarmState(string reason)
    {
        if (_cachedAlarmState.ActiveSeverity > 0)
        {
            _cachedAlarmState = new AlarmStateDto();
            PublishStateChangedInternal("device_normal", "alarm",
                reason, "所有告警已清除");
        }
    }

    private AlarmStateDto EvaluateAlarmState(AlarmStateDto current, DetectionAlertDto alert)
    {
        // 更新最近告警列表（保留最近 5 条）
        var recent = new List<string>(current.RecentAlarmTypes) { alert.AlarmType };
        if (recent.Count > 5) recent = recent.Skip(recent.Count - 5).ToList();

        return new AlarmStateDto
        {
            ActiveSeverity = alert.Severity > 0 ? alert.Severity : 0,
            ActiveAlarmCount = alert.Severity > 0 ? current.ActiveAlarmCount + 1 : 0,
            LastAlarmTime = alert.Timestamp,
            RecentAlarmTypes = recent
        };
    }
}
```

### 3.7 DetectionPublisherService 改造

```csharp
public void OnAlertReceived(DetectionAlertDto alert)
{
    // 写入内部 channel（保留原有告警发布逻辑）
    _channel.Writer.TryWrite(alert);

    // === 新增：触发告警状态变更 ===
    _systemStateService.UpdateAlarmState(alert);
}
```

## 五、收益分析

| 维度 | 改造前 | 改造后 |
|---|---|---|
| **Locality** | 告警和状态变更逻辑完全分离，问题追踪需跨模块 | 告警评估和状态转换集中在一个位置 |
| **Leverage** | 远端订阅方需跨两个 MQTT 主题关联 | 只需订阅 `events/state_changed` 即可获知设备完整健康状态 |
| **可观测性** | 设备是否处于告警状态不可查询 | `SystemStateDto.Alarm` 提供完整的告警状态快照 |
| **向后兼容** | — | `detection/alerts` 主题保留，不破坏现有告警消费者 |

## 六、风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|---|---|---|---|
| 告警风暴导致状态变更事件泛滥 | 中 | 中 | 仅在严重度**变化**时推送（相同严重度的连续告警不重复推送） |
| 告警状态与实际情况不同步（假阳性） | 低 | 中 | 支持手动清除 + 超时自动降级 |
| `SystemStateDto` JSON 体积增大 | 低 | 低 | `AlarmStateDto` 字段精炼，增量 < 500 字节 |
